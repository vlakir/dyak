# Spec: T008 — GUI для нетехнического пользователя (тонкий фронтенд над CLI)

**Статус:** Done (реализовано 2026-06-23, один PR `T008-gui`)
**Дата создания:** 2026-06-23
**Связанные документы:** `BACKLOG.md` (запись T008), `CONCEPT.md` §17
(этапы реализации, GUI как финал), `DECISIONS.md` (ADR T013 `--progress-json`,
ADR T006 подстановка по заголовкам), `src/dyak/cli.py` (контракт команд),
`src/dyak/progress.py` (JSONL-контракт прогресса), будущий ADR T008
(выбор GUI-стека + контракт CLI↔GUI), зависимая задача T010 (Windows-сборка).

---

## 1. Overview

Сейчас dyak — только CLI: кадровик должен открыть терминал, вспомнить
команду `generate` и набор флагов (`--table`, `--template`, `--out`,
`--filename`, `--config`, `--sheet`, `--pdf`). Для нетехнического
пользователя это барьер. T008 — **финальная «вишенка на торте»**:
графическая оболочка в привычном оконном окружении, через которую
кадровик выбирает шаблон, таблицу, папку назначения и значимые опции,
жмёт «Сгенерировать» и видит понятный прогресс и результат — **без
терминала**.

Ключевой архитектурный принцип: GUI — **тонкий фронтенд над CLI**. Он
не дублирует доменную логику, а вызывает `dyak` через `subprocess` и
показывает его вывод. Ядро (`src/dyak`) остаётся единственным источником
истины; GUI — это «лицо» поверх уже стабильного CLI (этапы 0–4 + reverse,
релизы 0.1.0/0.2.0).

## 2. User Stories / Сценарии использования

- **Как кадровик без навыков терминала**, я хочу выбрать шаблон, таблицу
  и папку через файловые диалоги и нажать одну кнопку, чтобы получить
  пачку готовых документов, ни разу не открыв консоль.
- **Как кадровик**, я хочу видеть **живой прогресс** при генерации
  тысяч документов (бар + счётчик `N/Total`), чтобы быть уверенным, что
  программа не зависла.
- **Как кадровик**, я хочу при ошибке (нет колонки, битый шаблон, нет
  LibreOffice для PDF) видеть **понятное сообщение в окне**, а не
  трейсбек и не молчание.
- **Как кадровик**, я хочу **встроенный офлайн-help** по разделам
  (что такое шаблон, как размечать падежи, что делает каждая опция),
  чтобы разобраться без внешней документации и без интернета.
- **Как кадровик, начинающий с нуля**, я хочу кнопкой создать стартовый
  набор (как `dyak init`), чтобы получить пример шаблона/таблицы/конфига
  и оттолкнуться от него.

## 3. Functional Requirements

- **ДОЛЖНА** предоставлять окно с выборщиками (файловые диалоги) для:
  шаблона (`.docx`), таблицы (`.xlsx`), папки назначения (`--out`).
- **ДОЛЖНА** принимать только обязательное: таблицу, шаблон, папку
  результата. Опции `--filename` / `--sheet` / `--config` / `--pdf` из окна
  убраны по решению Разработчика 2026-06-23 (см. Resolved) — генерация
  всегда на значениях по умолчанию (имя по колонкам ФИО, активный лист,
  `./dyak.yaml` если есть рядом). Все эти опции доступны в CLI.
- **ДОЛЖНА** запускать генерацию **через `subprocess`** к `dyak`
  (логика не продублирована в GUI), с флагом `--progress-json`, и
  показывать живой прогресс-бар + счётчик `N/Total`, потребляя JSONL-
  события из stderr (`start`/`progress`/`done`/`error`).
- **ДОЛЖНА** выполнять генерацию **в фоне** — окно остаётся отзывчивым
  на тысячах строк (не «замерзает»).
- **ДОЛЖНА** по завершении показывать **человекочитаемый результат**
  (сколько документов, куда) или **понятную ошибку** (текст из stderr
  CLI, без трейсбека); по коду возврата CLI отличать успех от ошибки.
- **ДОЛЖНА** содержать **встроенный офлайн-help** по разделам, доступный
  из окна (без сети).
- **ДОЛЖНА** предоставлять отмену генерации (кнопка «Отмена» убивает
  subprocess); уже записанные файлы остаются, GUI честно сообщает
  «остановлено на N/Total» (Resolved E2).
- **НЕ ДОЛЖНА** дублировать доменную логику ядра (склонение, чтение
  xlsx, рендер docx) — только вызывать CLI.
- **НЕ ДОЛЖНА** требовать интернета ни для работы, ни для help.
- **НЕ ДОЛЖНА** выносить в окно команды check/init/reverse — базовый
  набор для нетехнического пользователя — только generate + справка
  (Resolved G2, пересмотр 2026-06-23). Эти команды доступны в CLI.

## 4. Success Criteria

- Нетехнический кадровик проходит путь «открыл окно → выбрал
  шаблон/таблицу/папку → нажал Сгенерировать → получил документы» **без
  терминала** и без чтения внешней документации.
- При пакете в несколько тысяч строк виден живой прогресс (бар +
  `N/Total`), окно не «замерзает» (UI отзывчив во время генерации).
- При ошибке CLI (ненулевой exit-код) GUI показывает понятное
  сообщение в окне (не трейсбек, не тихий выход).
- Встроенный help открывается офлайн и покрывает основные разделы.
- Приложение запускается на Linux в оконном окружении (Windows —
  желательно, через T010).
- 4 проверки зелёные, coverage ≥ 80% на `src/` (стратегия покрытия GUI —
  см. Clarify, тестируемая логика-обёртка отделена от виджетов).
- В `examples/` добавлен **новый** пример запуска GUI со своим
  mini-`README` (правило проекта: GUI — новый способ использования).

## 5. Key Entities

- **Главное окно** — форма с полями ввода (пути, опции) и кнопками
  (Сгенерировать, Help, опц. init/check/reverse).
- **Обёртка вызова CLI** (`runner`) — чистый, тестируемый слой: собирает
  argv из значений формы, запускает `subprocess`, читает stdout/stderr,
  парсит JSONL-события прогресса, возвращает структурированный результат
  (успех/ошибка, счётчики, текст). **Главная точка тестирования.**
- **Модель прогресса** — состояние бара (`done`/`total`/текущий файл),
  обновляемое из потока событий.
- **Контент help** — статические офлайн-тексты по разделам.

## 6. Assumptions & Constraints

- Пользователь — нетехнический кадровик; цель — ноль терминала.
- CLI стабилен (этапы 0–4 + reverse, 0.1.0/0.2.0); контракт прогресса
  (`--progress-json`, JSONL в stderr) уже готов (T013).
- `dyak` доступен как исполняемая команда / модуль (`python -m dyak`);
  как именно GUI его находит — Clarify (D1).
- Python 3.14, `uv`, `src/`-layout; ruff(ALL)+mypy 0 ошибок;
  pytest coverage ≥ 80% на `src/`.
- Linux обязателен; Windows желателен (сборка — отдельная задача T010,
  PyInstaller). Выбор стека влияет на размер/сложность Windows-бандла.
- Локализация интерфейса — русский (пользователь русскоязычный);
  мультиязычность — см. Clarify (G4).

## 7. Out of Scope

- **Windows-сборка** (PyInstaller, инсталлятор, portable) — отдельная
  задача **T010**, не здесь.
- **Нагрузочное тестирование / профилирование** GUI и генерации — **T009**.
- **Локальный AI-режим** (T007 LLM / T014) в GUI — позже, когда фича
  ядра появится; GUI оставляет место под опцию, но не реализует.
- Переписывание/дублирование доменной логики в GUI.
- Веб-интерфейс, клиент-серверная архитектура — только локальное
  десктоп-приложение.
- Редактор шаблонов/таблиц внутри GUI — только выбор готовых файлов.

## Analyze (заполняется Claude, 2026-06-23)

### 🔴 Critical (фиксим до кода)

- **🔴-1. Колёса PySide6 под Python 3.14 — РАЗРЕШЕНО спайком.** Риск
  «как natasha» снят: PySide6 6.11.1 ставится и импортируется на 3.14.5.
- **🔴-2. Headless-тесты GUI в CI.** Любое создание `QApplication` без
  дисплея падает. Решение: smoke-тесты виджетов под
  `QT_QPA_PLATFORM=offscreen` (env в conftest/CI); основное покрытие —
  на `runner.py` без Qt вообще. Зафиксировать в conftest и в CI workflow.
  **Доп. (выявлено на первом прогоне CI, 2026-06-23):** даже `import
  PySide6.QtWidgets` под offscreen дёргает `libEGL.so.1`/`libGL`, которых
  нет на голом ubuntu-раннере → в `ci.yml` добавлен шаг `apt-get install
  libegl1 libgl1 libxkbcommon0 libdbus-1-3` перед `uv sync`.

### 🟡 Warning (обсудить/учесть)

- **🟡-1. Парсинг JSONL прогресса устойчиво к мусору.** stderr CLI
  смешивает (а) JSONL-события (`--progress-json`), (б) человекочитаемые
  логи (`logger.warning`, напр. «не распознаны колонки ФИО»),
  (в) при ошибке — `Ошибка: …` от typer. Парсер обязан: пытаться
  `json.loads` каждой строки, не-JSON — складывать в лог-панель (это
  warning'и, полезные кадровику), не падать. Не предполагать, что весь
  stderr — JSON.
- **🟡-2. Поток событий из subprocess без блокировки UI.** Читать
  stderr построчно в фоне (QThread/QProcess.readyReadStandardError) и
  слать сигналы в UI-поток; нельзя читать в main-потоке (заморозит
  окно). Через `QProcess` это штатно (сигналы уже в event-loop) —
  предпочесть его «голому» subprocess+QThread, чтобы не городить мост
  потоков. Решить в реализации (QProcess vs subprocess+QThread).
- **🟡-3. Отмена и недописанный файл.** Kill subprocess может оставить
  частично записанный docx последней строки. Допустимо (по E2 не
  откатываем), но в сообщении об отмене честно сказать «остановлено на
  N/Total, файлы до N сохранены». Не выдавать отмену за успех.
- **🟡-4. Кодировки и пути с пробелами/кириллицей.** argv собираем
  списком (не строкой шелла) → пробелы/кириллица в путях безопасны.
  stderr читать как UTF-8 (CLI пишет `ensure_ascii=False`). Зафиксировать
  `encoding='utf-8'` при чтении процесса.
- **🟡-5. `--progress-json` только у `generate`.** check/init/reverse
  прогресса не эмитят (быстрые или разовые). Для них GUI просто ждёт
  завершения (индетерминированный «busy»-индикатор), показывает stdout
  как отчёт. Не ждать JSONL там, где его нет.

### 🟢 Note (к сведению)

- **🟢-1. `PySide6-Essentials` вместо `PySide6`.** Рантайму нужны только
  QtWidgets/QtCore — зависеть от essentials, не тащить addons (~167 МБ).
  Влияет на размер T010-бандла; сам функционал не страдает.
- **🟢-2. Зависимость — runtime, не dev.** GUI — часть продукта (финал
  CONCEPT), не инструмент разработки → `uv add PySide6-Essentials`
  (основная группа). Запуск ядра-CLI без GUI остаётся возможным
  (PySide6 импортируется только в `dyak.gui`, не в ядре).
- **🟢-3. Граница «тонкого фронтенда».** Единственное место с импортом
  PySide6 — пакет `dyak.gui`. Ядро (`cli`, `render`, `io`, …) о Qt не
  знает. Так CLI-only-сценарий и тесты ядра не тянут Qt.
- **🟢-4. init без `--out` сложнее остальных.** init создаёт файлы в
  каталоге (`--dir`), не генерирует из таблицы. В GUI это отдельная
  форма (выбор каталога + `--force`), а не часть generate-формы.
- **🟢-5. examples/ — новый пример.** GUI — новый способ использования →
  создать `examples/gui/` со своим mini-README (скрин/инструкция
  запуска `python -m dyak.gui`), не править CLI-примеры (правило
  проекта).
