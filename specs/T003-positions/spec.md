# Spec: T003 — Этап 2 «Должности» (склонение через pymorphy3)

**Статус:** Done
**Дата создания:** 2026-06-22
**Связанные документы:** `CONCEPT.md` §7.2–7.4, §5.3; `DECISIONS.md`
(ADR 2026-06-21 «Стек: два движка склонения»); `specs/T002-fio/spec.md`
(порт `Declinable`, падежные фильтры).

## 1. Overview

Научить dyak склонять должности (и произвольные существительные/фразы) по
падежам — пословно через `pymorphy3`. Кадровик пишет
`на должность {{ Должность | рд }}` и получает «на должность директора».
Сложные/несогласуемые фразы закрываются словарём `overrides` с наивысшим
приоритетом. Это снимает T002-ограничение, где `{{ Должность | рд }}`
возвращал строку без изменений.

## 2. User Stories

- Как кадровик, я хочу `на должность {{ Должность | вн }}` →
  «на должность директора», без падежных колонок в таблице.
- Как кадровик с фразой, которую движок берёт неидеально («заместитель
  генерального директора»), я хочу задать формы вручную в `dyak.yaml`
  и чтобы они победили автоматику.
- Как пользователь больших таблиц, я хочу, чтобы повторяющиеся должности
  не разбирались заново на каждой строке (кеш).

## 3. Functional Requirements

- ДОЛЖНА: склонять должность по 6 падежам **пословно** (`pymorphy3`):
  токенизация по пробелам и дефису, разбор каждого слова, `.inflect`,
  сборка обратно с сохранением разделителей.
- ДОЛЖНА: для роли `position` класть в контекст склоняемый объект
  `Position` (`Declinable`) — падежные фильтры T002 начинают его склонять.
- ДОЛЖНА: проверять `overrides.position` **до** движка; заданные падежи
  берутся из override, остальные — из движка (частичный override).
- ДОЛЖНА: инициализировать `MorphAnalyzer` один раз (общий с автоопределением
  пола T002) и кешировать результат склонения фразы (`lru_cache` по
  `(text, case)`).
- ДОЛЖНА: оставлять несклонившийся/неразобранный токен как есть (без падения).
- ДОЛЖНА: сохранять регистр первого слова при «капс»-записи? — НЕТ, см.
  Out of Scope (должности в таблицах обычно строчные; регистр — задача ФИО).
- НЕ ДОЛЖНА: пытаться синтаксически согласовывать прилагательные с
  существительными в сложных фразах сверх пословного склонения (R1) —
  для этого `overrides`.

## 4. Success Criteria

- «директор» → рд «директора», дт «директору», вн «директора», тв
  «директором», пр «директоре».
- «главный бухгалтер» → рд «главного бухгалтера» (пословно согласовано в
  типовом случае); «инженер-программист» → рд «инженера-программиста».
- `overrides.position` перекрывает движок: для «заместитель генерального
  директора» заданные формы выводятся дословно, ДО pymorphy.
- Частичный override: указан только `дт` → остальные падежи из движка.
- `MorphAnalyzer` создаётся один раз; повторная должность не парсится
  заново (кеш).
- 4 проверки зелёные, coverage ≥ 80%. `examples/` показывает склонение
  должности.

## 5. Key Entities

- **PymorphyInflector** — обёртка над общим `MorphAnalyzer`: `inflect(text,
  case)` пословно, с `lru_cache`. Род не нужен (существительные склоняются
  по падежу, не по полу человека).
- **Position** — `Declinable` должность: текст + движок + словарь
  override-форм для этого текста. `inflect(case)`: override → движок.
  `__str__` → именительный (исходный текст).
- **morph.get_analyzer** — общий кешированный `MorphAnalyzer` (вынесен из
  `gender.py`, чтобы движок пола и должностей делили один экземпляр — DRY).

## 6. Assumptions & Constraints

- Должности в таблице — в именительном, строчными (типовая кадровая
  практика). Регистр не реставрируем (в отличие от ФИО).
- `pymorphy3` + `pymorphy3-dicts-ru` уже в зависимостях (добавлены в T002).
- Детерминированность: берём первый разбор `parse(word)[0]` — стабилен на
  одном входе.
- Кеш `lru_cache` живёт в пределах процесса; на больших таблицах с
  повторами даёт основной выигрыш (§7.4, §15).

## 7. Out of Scope

- Согласование прилагательных в сложных фразах сверх пословного (R1) —
  закрывается `overrides`, не алгоритмом.
- Сохранение регистра должности (UPPERCASE) — не требуется кадровым
  сценарием; при необходимости — отдельная задача.
- `overrides.fio` для несклоняемых фамилий — petrovich и так справляется
  (T002); общий механизм override для ФИО при необходимости — отдельно.
- Команда `check`, `согл`, `StrictUndefined` — T004.

## Clarify (ответы — решения ВрИО Гвидо, ночной прогон 2026-06-22)

### Resolved

- **Q: Нужен ли род для склонения должности?**
  A: Нет. Существительные/фразы склоняются по падежу; род человека не
  влияет на «директора/бухгалтера». `PymorphyInflector.inflect(text, case)`
  без gender — проще и честнее. (Отклонение от Inflector-протокола
  CONCEPT §6, где `gender` в сигнатуре — там он для общности; для
  должностей не используется.)

- **Q: Ключ кеша — `(text, case, gender)` по CONCEPT §7.4 или `(text,
  case)`?**
  A: `(text, case)` — раз род не участвует. Зафиксировать в ADR-заметке.

- **Q: Где и как матчится override — по сырому тексту или нормализованному?**
  A: по нормализованному ключу (трим, схлопывание пробелов, нижний
  регистр) — как `overrides` в CONCEPT §7.3 («нормализованный исходный
  текст, регистронезависимо»). Значение — `CaseForms` (рус. сокращения).

- **Q: Токенизация — только пробел или и дефис?**
  A: пробел И дефис (с сохранением разделителей), чтобы
  «инженер-программист» склонялось по обеим частям.

- **Q: Регистр должности?**
  A: не реставрируем (Out of Scope). Должности строчные в таблицах.

## Analyze (Гвидо)

- 🟢 **N1.** Падежный фильтр уже маршрутизирует `Declinable` (T002) — для
  `{{ Должность | рд }}` достаточно положить `Position` в контекст под
  ключ роли `position`. Фильтры не трогаем.
- 🟢 **N2.** `MorphAnalyzer` уже создаётся в `gender.py` (T002). Чтобы не
  плодить второй экземпляр (DRY — главный принцип Разработчика), выношу
  доступ в `inflection/morph.py` и переключаю `gender.py` на него. Это
  правка T002-кода, но строго в духе scope (убираем дублирование,
  вводимое вторым потребителем pymorphy).
- 🟡 **W1.** Пословное склонение рассогласует «заместитель генерального
  директора» в косвенных (зависимый род. оборот склоняется) — известное
  R1. Закрываем `overrides` (приоритет ДО движка). Покрыть тестом
  override + тестом, демонстрирующим ограничение.
- 🟢 **N3.** Именительный (`Case.NOMN`): pymorphy `inflect({'nomn'})` ≈
  исходный текст, но дешевле и безопаснее вернуть текст как есть (или
  override `им`).
- 🟢 **N4.** `lru_cache` на свободной функции `(text, case)` — не на методе
  (чтобы не держать ссылку на self/инфлектор). Анализатор берётся внутри
  через кешированный `get_analyzer()`.
- 🟢 **N5.** `Position` хранит срез override-форм только для своего текста
  (строится в контексте по `overrides.position[ключ]`), поэтому
  `inflect` не зависит от глобального словаря.

Проход чистый — реализуем по фазам.

4. **examples.** Шаблон: должность через падежный фильтр + пример
   `overrides` в `dyak.yaml`; прогон.
