# dyak

**Дьяк** — CLI-инструмент пакетной генерации однотипных кадровых
документов. На вход: таблица персональных данных (xlsx) и шаблон Word
(docx). На выходе: набор документов, по одному на строку таблицы, с
**автоматическим склонением русских ФИО и должностей по падежам** в
зависимости от места подстановки в шаблоне. Разметка шаблонов —
полностью на русском (пользователи — кадровики, не технические
специалисты).

Полное изначальное видение и ТЗ — в [`CONCEPT.md`](CONCEPT.md).
Статус: **0.3.0** (2026-06-24). Реализованы этапы 0–4 и обратная сборка:
генерация со склонением ФИО / должностей / воинских и служебных званий,
автоопределение пола, сухой прогон `check`, `init`-scaffold, экспорт в
PDF, индикатор прогресса, обратная сборка шаблона из готового документа
(`reverse`), распознавание ролей колонок по содержимому (нестандартные
заголовки без `aliases`), учёт фильтра/скрытых строк xlsx. Появилась
**графическая оболочка** для нетехнического пользователя (`python -m
dyak.gui`, T008) и **сборка Windows-бинарников** (инсталлятор + portable)
через GitHub Actions по git-тегу (T010). Следующий этап — `BACKLOG.md`:
нагрузочное тестирование (T009).

## Быстрый старт

Менеджер зависимостей и окружения: **`uv`** (выбран при
`dreamteam init`).

```bash
uv sync                       # поставить зависимости

# графическая оболочка для нетехнического пользователя (T008)
PYTHONPATH=src uv run python -m dyak.gui

# старт с нуля: выложить пример конфига, шаблона и таблицы
PYTHONPATH=src uv run python -m dyak init --dir ./my_project

# генерация документов (склонение по падежным фильтрам в шаблоне)
PYTHONPATH=src uv run python -m dyak generate \
  --table employees.xlsx \
  --template order.docx \
  --out ./output \
  [--filename "Приказ_{{ Номер_приказа }}_{{ Фамилия }}.docx"] \
  [--config dyak.yaml] \
  [--sheet "Лист1"] \
  [--progress-json] \  # машиночитаемый прогресс (JSONL) в stderr — для GUI
  [--pdf]              # дополнительно сконвертировать вывод в PDF
```

На больших пакетах `generate` в интерактивном терминале показывает живой
прогресс-бар; `--progress-json` дополнительно эмитит построчные
JSON-события в stderr (контракт для GUI), оставляя stdout чистым. Флаг
`--pdf` кладёт рядом с каждым docx его PDF через LibreOffice headless
(нужен установленный LibreOffice; иначе — понятная ошибка, не трейсбек).

Теги в docx-шаблоне ссылаются **напрямую на заголовки колонок** таблицы:
`{{ Фамилия }}`, `{{ Должность }}`, `{{ Дата_начала }}` (пробел в
заголовке → подчёркивание; забытый пробел Дьяк чинит сам + warning).
Стандартные кадровые колонки распознаются по названию, поэтому
`dyak.yaml` **опционален** (только нестандартные заголовки → роли через
`aliases` и ручные `overrides`). Имя выходного файла — флаг `--filename`
(по умолчанию по колонкам ФИО). Колонка «Пол» не нужна: пол определяется
автоматически (T002). Отклонение от `CONCEPT.md` §6, §8–§9 зафиксировано
в `DECISIONS.md` (ADR 2026-06-21).

## Сборка под Windows и релиз (T010)

Готовые бинарники GUI «Дьяк» под Windows собирает GitHub Actions
(`.github/workflows/build-windows.yml`) — Python ставить пользователю не
нужно. На релиз выпускаются два артефакта:

- `dyak-X.Y.Z-setup.exe` — инсталлятор (Inno Setup), установка в профиль
  пользователя без прав администратора, ярлык на рабочий стол;
- `dyak-X.Y.Z-portable.exe` — портативная версия одним файлом (без
  установки и без вложенных папок; стартует чуть дольше из-за распаковки).

**Релиз — постановкой git-тега** `vX.Y.Z` на milestone-коммит в `main`
(версия должна совпадать с очередной `[N.M.0]` в `CHANGELOG.md`):

```bash
git tag v0.3.0 && git push origin v0.3.0
```

Тег запускает workflow, собирает оба артефакта и создаёт GitHub Release.
Для проверочного прогона без релиза — ручной запуск (`workflow_dispatch`)
во вкладке Actions. Локальная сборка (для отладки spec):

```bash
uv sync --group build
# → dist/dyak/ (onedir для инсталлятора) + dist/dyak-portable (onefile)
uv run pyinstaller packaging/dyak.spec --noconfirm --clean
```

Бандл — единый exe (`packaging/dyak.spec` поверх `dyak._app_entry`): без
аргументов открывает окно, с аргументами работает как CLI-ядро, поэтому
GUI вызывает ядро тем же exe через subprocess и в собранном виде.

## Проверки перед push

```bash
uv run ruff check .
uv run ruff format --check .
uv run mypy <путь к коду>
```

Все три должны проходить с 0 ошибок. Обходные манёвры (`# noqa`,
`# type: ignore`, расширение `ignore`-секции) — только по согласованию.

## Структура проекта

- `src/` — корень исходников.
- `CONCEPT.md` — изначальное видение проекта (immutable).
- `DECISIONS.md` — архитектурные решения с обоснованиями (ADR-Lite).
- `BOARD.md` — рабочая Kanban-доска (To Do / Doing / Done).
- `BACKLOG.md` — парковка идей и побочных находок.
- `CHANGELOG.md` — журнал заметных изменений.
- `specs/` — спецификации крупных фич.
- `CLAUDE.md` — проектные правила для Claude (Claude Code).
