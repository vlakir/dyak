# Backlog

Парковка идей, побочных находок и «надо бы потом починить».

**Правило:** если в процессе работы над текущей задачей Claude или
Разработчик замечают что-то постороннее — оно идёт сюда, а не в текущий
коммит. Это защищает от расползания scope.

Это **не формальный таск-трекер** со сроками и метриками — это парковка
идей. Но **порядок имеет значение**: сверху — то, что планируется
ближайшим, ниже — менее срочное (FIFO по умолчанию, можно поднимать
приоритетное наверх). Когда из бэклога что-то берётся в работу — оно
вырастает в задачу или спеку (`specs/T<NNN>-…`) и удаляется отсюда.

## Формат

`- **T<NNN>** — [<дата находки>] <короткое описание> — <опционально: контекст / откуда всплыло>`

ID присваивается при создании; новый = `max(существующих T-ID в
BACKLOG.md, BOARD.md и CHANGELOG.md) + 1`. ID не переиспользуется
и сохраняется при перетекании задачи между BACKLOG и BOARD; после
релиза задача переходит в `CHANGELOG.md` (с тем же T-ID), что
гарантирует уникальность между релизами.

- **T020** — [2026-06-25] **Приоритет роли: имя/alias колонки важнее
  контент-догадки.** Колонка «Подразделение» (без синонима-заголовка) по
  содержимому классифицируется как `position` (`columns.py:217` — любой
  текст без имён-грамем → POSITION) и, идя раньше по порядку, **забирает
  роль `position` у «Должности»** (точный синоним `должность→position`,
  `columns.py:64`). «Должность» затем игнорируется (`recognize`,
  `columns.py:310` — first-come-first-served) и **не склоняется**
  («командир» вместо «командиру»). Аналогично «№ п/п» (контент →
  `personal_number`) перебивает «л/н». Корень: при конфликте за одну роль
  претендент **по имени/alias** должен побеждать претендента **только по
  содержимому**, независимо от порядка колонок. Контент-перебивание
  заголовка (T016, `_decide_role`) оставить только для единогласного
  расхождения внутри одной колонки, не для кражи роли у другой колонки.
  Воспроизводится в ядре И бандле (обе платформы). Контейнерный фикс в
  `columns.py` (приоритет источника при коллизии роли) + регрессионный
  тест на пример приказа. ADR при взятии (правило приоритета).
  Acceptance: при наличии колонки с синонимом/alias под роль — она
  получает роль даже если другая колонка раньше по порядку угадала ту же
  роль по содержимому; «Должность» в примере склоняется («командиру»);
  warning о коллизии сохраняется для реально лишних колонок; старые
  таблицы не регрессируют (тест T006/T016 зелёные).

- **T022** — [2026-06-25] **Качество склонения многословных фраз +
  роль «Подразделение».** Фраза-склонение должности/подразделения
  разваливает словосочетания: «рота связи»|вн → «роту **связь**» (надо
  «роту связ**и**» — родительный хвост замораживается, как у званий в
  T016 фазы B); «взвод охраны» → «взвод охран**у**»; «1 мотострелковый
  батальон»|вн → «мотострелков**ого** батальон» (не учтена
  одушевлённость: батальон неодушевл., вн=им). Плюс — у «Подразделение»
  **нет своей роли**: даже после T020 без роли оно пройдёт несклонённым
  (фильтр падежа склоняет только `Declinable`, `filters.py:51`), а
  пользователь хочет его в нужном падеже (`{{ Подразделение|вн }}`).
  **Крупная фича / правка семантики склонения → полный ритуал
  spec/clarify/analyze при взятии.** Открытые вопросы для clarify:
  выделять ли отдельную роль «subdivision»/«unit» или общий механизм
  «склоняемая текстовая колонка»; как определять вершину фразы и
  замороженный родительный хвост (морфоанализ vs эвристика); одушевлённость
  для винительного; взаимодействие с reverse (T007). Зависит от T020
  (сначала корректная маршрутизация ролей). Acceptance (черновой, уточнить
  в спеке): «рота связи»|вн → «роту связи», «взвод охраны»|вн → «взвод
  охраны», «1 мотострелковый батальон»|вн → «1 мотострелковый батальон»;
  «Подразделение» склоняется в запрошенный падеж; регресс-фикстура из
  примера приказа даёт грамматически корректный текст по всем 4 строкам.

- **T023** — [2026-06-25] **Windows-GUI: кракозябры в окне лога (кодировка
  subprocess↔GUI).** В GUI на Windows окно лога показывает mojibake вместо
  русских букв (например, итог «Готово: N документ(ов)» и warning'и ролей).
  Корень: GUI читает поток `QProcess` и декодирует его как **UTF-8**
  (`main_window.py:200,208` — `decode('utf-8', errors='replace')`), а
  дочерний Python на русской Windows пишет stdout/stderr через пайп в
  **cp1251** (ANSI-кодировка локали, т.к. поток не консоль и не задан
  `PYTHONUTF8`/`PYTHONIOENCODING`). Несовпадение → кракозябры. Воспроизводится
  **только на Windows** (на Linux дефолт UTF-8, потому не всплыло). Зависит
  от T008 (GUI) / T010 (Windows). **Целевой фикс — на границе GUI→ядро:**
  выставить дочернему процессу окружение `PYTHONUTF8=1` (или
  `PYTHONIOENCODING=utf-8`) через `QProcess.setProcessEnvironment` перед
  `start()`, чтобы ядро эмитило UTF-8 и совпадало с UTF-8-декодом GUI. НЕ
  трогать кодировку прямого CLI (вывод в cp1251-консоль Windows оставить как
  есть — правим только подпроцесс под GUI). Проверить, что `PYTHONUTF8`
  honor-ится embedded-интерпретатором в PyInstaller-onefile (если нет —
  fallback: `sys.std*.reconfigure(encoding='utf-8')` только когда ядро
  запущено из GUI, по флагу/переменной). **Естественно батчится с T021**
  (оба только Windows, оба требуют проверки на `windows-latest` и идут в
  0.3.1). Acceptance: в Windows-GUI окно лога показывает корректные русские
  буквы (итоги/warning'и/ошибки); JSONL-прогресс по-прежнему парсится;
  прямой CLI на Windows не регрессирует; Linux-GUI без изменений.
