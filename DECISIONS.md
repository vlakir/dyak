# Architecture Decisions

ADR-Lite: компактный лог архитектурных решений с обоснованиями.
Цель — через полгода можно ответить на вопрос «а почему мы тогда так
сделали?», не реконструируя контекст по коммитам.

## Формат

Каждое решение — короткий блок:

- **Дата** — когда принято.
- **Решение** — что решили (1 строка).
- **Контекст** — какая задача / ограничение к нему привели.
- **Альтернативы** — что рассматривали и почему отвергли.
- **Последствия** — что это нам теперь даёт и чего лишает.

Решения не редактируются после фиксации. Если решение пересмотрено —
добавляется новый блок со ссылкой на старый, старый помечается как
«Заменено решением от <дата>».

### 2026-06-23 — Фильтр/скрытые строки xlsx: генерация только из видимых, распознавание по всем; загрузка без `read_only` (T019)

- **Решение:** при пакетной генерации документы формируются **только из
  видимых** строк таблицы (не скрытых автофильтром или вручную), а контентное
  распознавание ролей (T016) использует образцы со **всех** строк, включая
  скрытые. Реализация в `io/excel.py`:
  - книга грузится **без** `read_only` (`load_workbook(path, data_only=True)`);
  - строки собираются с номером листа (`enumerate(..., start=2)`); скрытость
    проверяется `ws.row_dimensions[r].hidden`;
  - образцы для `recognize` пополняются по всем непустым строкам; в `people`
    (на генерацию) попадают только строки с `hidden is False`;
  - число пропущенных скрытых строк логируется (`logger.info`).
- **Контекст:** запрос Разработчика 2026-06-23 — если в таблице включён
  фильтр, генерировать только по «открытым» строкам, но при сопоставлении
  использовать всю информацию (в т.ч. из «закрытых»). Так фильтр управляет
  выборкой документов, не обедняя распознавание.
- **Альтернативы:**
  - **Оставить `read_only=True`** — невозможно: в read-only режиме openpyxl не
    отдаёт `row_dimensions`/`auto_filter` (проверено), скрытость не прочитать.
  - **Парсить XML `<row hidden="1">` отдельно**, сохранив read_only — отвергли:
    сложнее и хрупче ради экономии памяти, не нужной на кадровых объёмах.
  - **Вычислять фильтр по критериям `auto_filter`** (а не по флагу `hidden`) —
    отвергли: openpyxl не применяет фильтр; реальная скрытость живёт во флаге
    `hidden`, который Excel ставит при применении фильтра/ручном скрытии (для
    нас фильтр и ручное скрытие неразличимы и трактуются одинаково).
- **Последствия:** потерян `read_only`-оптимизация чтения (ставилась ради
  памяти на больших батчах) — на кадровых таблицах (сотни–тысячи строк)
  незаметно; экстремальные объёмы (100k+) — забота нагрузочного этапа T009
  (там пересмотреть, если упрёмся). Эффект единый для `generate`/`check`/
  `reverse` (все читают через `read_table`); для `reverse --row N` нумерация
  теперь по видимым строкам (консистентно с генерацией). Пустые строки
  по-прежнему пропускаются раньше проверки скрытости (скрытая пустая строка
  не считается «пропущенной»).

### 2026-06-23 — Reverse: предупреждение о подписи/исполнителе по сумме совпадений имени (T018)

- **Решение:** reverse добавляет находку `signature_risk`, когда имя субъекта
  заменено тегами **в нескольких местах** (≥2). Подпись и блок исполнителя
  содержат фамилию/инициалы; при совпадении с субъектом строки они тоже
  тегируются, хотя там лицо обычно постоянное (тег лишний). Reverse не может
  знать, какое вхождение «настоящее», поэтому **только предупреждает** —
  кадровик правит руками. Счёт ведётся **по всей семье ФИО-кандидатов**
  (целое ФИО + фамилия/имя/отчество + три формы инициалов), помеченных флагом
  `Candidate.name_part`: тело документа и подпись-инициалами дают по одному
  совпадению на **разных** кандидатов, поэтому «>1 на одного кандидата» этот
  случай бы не поймал.
- **Контекст:** вопрос Разработчика 2026-06-22 — не примет ли reverse фамилию
  из подписи/исполнителя за место вставки. Эмпирически подтверждено: «слепого»
  режима у reverse нет (сопоставляет только значения строки через
  `find_spans` по формам кандидатов), поэтому **чужая** фамилия не тегируется
  и ничего не выдумывается (`not_found` при отсутствии). Реальный риск —
  только **совпадение фамилии** субъекта и подписанта/исполнителя; его и
  закрывает предупреждение.
- **Альтернативы:**
  - **Эвристика «зоны подписи/исполнителя»** (не сопоставлять после маркеров
    «Исп.»/таб-отступа+инициалы) — отложили: хрупко (формат подписи не
    стандартизован), риск пропустить/переусердствовать; предупреждение проще
    и честнее для best-effort инструмента.
  - **«×N» на одном кандидате** (как первая идея) — отвергли: не ловит
    тело-полное-ФИО + подпись-инициалами (разные кандидаты, по 1 разу); нужна
    сумма по семье `name_part`.
  - **Молча тегировать** (как было) — отвергли: подпись с тегом тихо
    разъедет имена при генерации для других людей.
- **Последствия:** новая категория отчёта `signature_risk` (раздел «Проверьте
  подпись/исполнителя»). Предупреждение **build-time** (одно на сборку
  шаблона) — может срабатывать и на легитимном повторе имени в теле, но это
  уместный разовый повод проверить, не шум на каждый документ. Должность/звание
  в подписи (тоже возможный повтор) сейчас не покрыты — `name_part` только про
  имя; расширение при необходимости.

### 2026-06-22 — Reverse склоняет звания: проводка `RankInflector` + `Rank` в isinstance кандидатов (T017)

- **Решение:** `dyak reverse` разворачивает колонку «Звание» в падежный тег
  (`{{ Звание | дт }}`) через тот же **decline-and-match**, что и ФИО/должности
  (T007). Два узких изменения:
  1. `reverse/candidates.py`: `Rank` добавлен в `isinstance(value, NamePart |
     Position | Rank)` склоняемых кандидатов — `_decline_forms` генерик по
     `Declinable`, поэтому 6 падежных форм и матчинг работают без новой логики.
  2. Проводка `rank_inflector`/`rank_overrides` через `cli.reverse_template` →
     `build_template` → `build_candidates` **и** `_verify_roundtrip` (round-trip
     строит контекст тем же `build_context`, поэтому тег воспроизводится и
     сверка не даёт ложного `roundtrip_mismatch`).
- **Контекст:** хвост T016 фазы B — там движок званий подключили к forward
  (`generate`/`check`), но в reverse (T007) не пробросили, и «Звание» оставалось
  обычной строкой (точное именительное совпадение, без падежей). Запарковано в
  ADR T016 фазы B, закрыто отдельной задачей T017.
- **Альтернативы:**
  - **Сделать `_declinable_candidates` генериком по `Declinable`** (вместо
    явного isinstance) — отвергли пока: явный список типов нагляднее и
    контролирует, что считается склоняемым кандидатом (ФИО/должность/звание);
    инициалы и целое ФИО уже имеют отдельную ветку.
  - **Decline-and-match по полному званию без заморозки хвоста** — не нужен:
    `Rank.inflect` уже даёт правильные формы (голова склоняется, хвост замирает),
    формы различимы по голове, омонимия головы → существующий `ambiguous`.
- **Последствия:** reverse теперь симметричен forward по званиям; `overrides.rank`
  подхватывается (формы из `build_context`). «Личный номер» (несклоняемый)
  по-прежнему матчится точным совпадением (плоская ячейка). T016 закрыт
  полностью (включая обратный путь).

### 2026-06-22 — Пустые ячейки: empty-aware чистка через Jinja `finalize`-маркер (T016, фаза C)

- **Решение:** пустая опциональная подстановка не оставляет в документе ни
  лишнего пробела, ни висячей пунктуации. Механизм **empty-aware**, в два шага:
  1. **Маркер пустых подстановок** — Jinja `finalize`-хук (`render/filters.py`,
     `EMPTY_MARKER` = символ Private Use Area `U+E000`) подменяет любой пустой
     результат `{{ … }}` маркером. Так пост-обработка **точно знает**, где было
     пустое значение, не угадывая по тексту.
  2. **Пост-обработка параграфа** (`render/engine.py`, `_clean_paragraph`) —
     маска «оставить/удалить» по склейке run'ов: снимает маркер; при пустом
     значении убирает **висячую завершающую** пунктуацию (`. ! ? …`) в конце
     параграфа, **сохраняя разделители** (`, ;`) и лейбл («Звание: ▮.» →
     «Звание:», «Оклад ▮, надбавка.» → «Оклад, надбавка.»); затем общая чистка
     — схлопывание кратных пробелов, удаление пробела перед закрывающей
     пунктуацией (`. , ; : ! ? » ) …`), обрезка краёв параграфа. Маркер всегда
     снимается до сохранения/сверки; в имени файла снимается отдельно
     (`render_filename`).
- **Контекст:** спайк (запись T016 в BACKLOG) показал 3 косметических артефакта
  от пустых ячеек (пробел перед пунктуацией, ведущий пробел). Разработчик
  (clarify C1) выбрал «убрать и точку» — это **условная логика по пустому
  полю**, которую нельзя надёжно сделать чисткой готового текста (потеря
  признака «было пусто» → слепое удаление точки портит легитимные предложения).
  Отсюда осознанное расширение фазы C до empty-aware рендера (согласовано
  2026-06-22). Фаза C зонтичной задачи T016; spec §3C/4C.
- **Альтернативы:**
  - **Чистка готового текста без маркера** (regex по «пробел+пунктуация») —
    отвергли: не отличает артефакт пустого поля от легитимной пунктуации,
    портит реальный текст.
  - **`{% if Поле %}…{% endif %}` в шаблоне** (docxtpl умеет) — отвергли как
    основной путь: нетехнический кадровик такого не напишет (CONCEPT — простота
    для кадровика); остаётся доступным вручную для сложных случаев.
  - **Удалять и лейбл «Звание:»** при пустом поле (clarify) — отвергли:
    dyak не знает границ лейбла; «Звание:» остаётся (решение Разработчика).
  - **Маркер обычным символом/пробелом** — отвергли: PUA `U+E000` гарантированно
    не встречается в кадровых текстах, ложных срабатываний нет.
- **Последствия:** все рендеры идут через `finalize` (новый параметр окружения)
  — маркер транзитный, живёт только между Jinja-рендером и `_clean_paragraph`,
  поэтому reverse round-trip (T007) и `check` не ломаются (обе стороны чистятся
  одинаково). Старый `_collapse_paragraph_spaces` заменён на `_clean_paragraph`
  (та же задача + висячая пунктуация + края, через единую маску по run'ам).
  Удаление висячей пунктуации ограничено **концом параграфа** (безопасно);
  лейбл-строки вида «Звание:» остаются (исчезновение всей строки — Out of Scope,
  условная логика). Завершает T016 (фазы A+B+C).

### 2026-06-22 — Склонение званий: голова + замороженный генитивный хвост, новый `Rank`/`overrides.rank` (T016, фаза B)

- **Решение:** воинские/служебные звания склоняет отдельный движок
  `RankInflector` (`inflection/rank.py`), а не пословный движок должностей.
  **Гибрид-эвристика** (решение Разработчика 2026-06-22):
  1. **Голова склоняется пословно** — токены до генитивного хвоста (так
     согласованное прилагательное склоняется вместе: «старший лейтенант» →
     «старшему лейтенанту»).
  2. **Хвост замораживается** — всё от маркера остаётся как в источнике
     (родительный): маркер = числовой токен («3 ранга»), слово с заглавной
     (имя собственное «Российской Федерации») или курируемое маркер-
     существительное (`_TAIL_MARKERS`: службы/юстиции/полиции/ранга/класса/
     флота/…), с откатом назад через согласованное прилагательное
     («медицинской службы» — замораживается и «медицинской»).
  3. **Дефис** — склоняется только последний сегмент токена
     («контр-адмиралу», «генерал-майору»); служебная приставка не трогается.
  - Новый доменный объект **`Rank`** (`Declinable`) с приоритетом ручного
    **`overrides.rank`** над движком — зеркало `Position`/`overrides.position`.
    Маршрутизация роли `rank` в `render/context.py` (`_add_rank`), проводка
    `rank_inflector`/`rank_overrides` через `build_context` в `generate` и
    `check` (там же `not_declined`-эвристика теперь покрывает звания).
  - Общий примитив склонения слова вынесен в `pymorphy_phrase.inflect_word`
    (DRY: используют и пословный движок фраз, и движок званий).
- **Контекст:** спайк 2026-06-22 (запись T016 в BACKLOG) показал, что
  пословный движок ломает составные звания — тащит несогласованный хвост в
  целевой падеж («майору медицинской **службе**», «капитану 3 **рангу**»,
  «майором полиц**ией**») и склоняет дефисные приставки («**контрам**-
  адмиралу»), теряя регистр имён собственных. Фаза B зонтичной задачи T016
  (spec `specs/T016-content-roles/spec.md`); распознавание колонки «Звание» —
  фаза A.
- **Альтернативы:**
  - **Пословный движок должностей as-is** — отвергли (ломает составные звания,
    см. спайк).
  - **Только `overrides.rank`** (без эвристики) — отвергли: кадровик писал бы
    формы руками для каждого составного звания; эвристика покрывает типовые
    автоматически (проверено на медслужбе/флоте/госслужбе/дефисных/имени
    собственном).
  - **Словарь готовых форм всех званий** — отвергли: огромный словарь вручную
    (армия/флот/МВД/юстиция/медслужба), неполнота = молчаливый промах;
    acceptance фазы B допускает «хотя бы явный механизм».
  - **POS-граница «замораживай после первого существительного»** — отвергли:
    pymorphy парсит «старший»/«контр» как NOUN, граница по части речи
    неверна; нужен маркер-список генитивного хвоста.
  - **Переиспользовать `Position`/`overrides.position`** — отвергли (B2,
    решение Разработчика): смешало бы семантику звание/должность в одном
    namespace оверрайдов и протащило бы head-only логику в должности.
- **Последствия:** составные звания склоняются автоматически; `Rank`
  подключён к контексту тем же портом `Declinable`, что ФИО/должности
  (фильтры не трогаются). `build_context`/`check_table` получили параметры
  `rank_*`. **Reverse (T007) звание пока НЕ склоняет** — `reverse_template`
  не передаёт `rank_inflector` (звание в reverse остаётся обычной строкой,
  как было; не регрессия) — decline-and-match по званиям запаркован
  (следующий шаг, вне scope фазы B). `_TAIL_MARKERS` пополняется по находкам.

### 2026-06-22 — Распознавание ролей колонок по содержимому: pymorphy-грамемы + словарь званий, жёсткий порог перебивания (T016, фаза A)

- **Решение:** роль колонки определяется не только по заголовку, но и **по
  содержимому ячеек** (`columns.py`, `infer_role`/`_classify_cell`), на **уже
  имеющемся** `pymorphy3` (0 новых зависимостей). Приоритет источников:
  1. **`aliases`** (`dyak.yaml`) — абсолютный верх, ручной override.
  2. **Заголовок-синоним** — статический словарь `_SYNONYMS`.
  3. **Содержимое** — мажоритарное голосование по ≤5 образцам каждой колонки:
     грамемы `Surn`/`Name`/`Patr` (по **максимальному score** разбора, не
     «есть ли грамема вообще» — иначе должности с фамилие-подобными словами
     уехали бы в surname) дают фамилию/имя/отчество/целое ФИО; словарь
     **вершинных маркеров званий** (`_RANK_MARKERS`) — роль `rank`; маска
     кода (цифры/буквы) — `personal_number`; маска даты исключает колонку из
     текстовых ролей; остаток текста без name-грамем → `position`.
  - **Жёсткий порог перебивания заголовка (🔴-1 спеки):** уверенный контент
    перебивает знакомый заголовок-синоним ТОЛЬКО когда (а) образцов ≥3 и
    **все** единогласны на другую роль, И (б) роль заголовка не подтверждена
    ни одним образцом. Иначе заголовок главнее, расхождение лишь логируется.
    Так ловится ошибка разметки, но нулевая регрессия T001–T007 защищена
    (без образцов `recognize` вообще не трогает pymorphy — чистый header-only
    путь T006).
  - **Неоднозначность → лучшая догадка + warning** (решение Разработчика):
    неуверенная колонка получает наиболее вероятную роль и `logger.warning`
    (+ задел под строку отчёта `check`), не падает и не молчит.
  - Новые роли `rank`/`personal_number` в `Role` (`config.py`). В фазе A они
    только **распознаются**; склонение званий (вершина + замороженный хвост) —
    отдельная фаза B (`overrides.position` уже сейчас закрывает составные
    звания вручную). `personal_number` — несклоняемый, подставляется буквально.
- **Контекст:** этап из расширенной записи T016 (`BACKLOG.md`); кадровые
  таблицы редко имеют стандартные заголовки («Сотрудник», «Наименование
  должности», «Фамилия, имя, отчество»), а ручной `aliases` — барьер для
  нетехнического пользователя (та же мотивация, что и ADR T006). T016 ведётся
  одной задачей в три фазы (A: распознавание / B: склонение званий / C: чистка
  пробелов), каждая — отдельный PR под зонтиком (решение Разработчика
  2026-06-22); spec `specs/T016-content-roles/spec.md`.
- **Альтернативы:**
  - **Только заголовок-синоним + `aliases`** (как до T016) — отвергли: не
    решает нестандартные заголовки без ручного конфига.
  - **Заголовок всегда главнее контента** (исходный гибрид) — отвергли по
    решению Разработчика в пользу «контент может перебить при единогласии»
    (ловит ошибку разметки), но с жёстким порогом против ложных срабатываний.
  - **ML/LLM-классификатор колонок** — отвергли: детерминированной эвристики
    на pymorphy достаточно для кадровых ролей, без веса моделей и облака.
  - **Распознавание «Звание» грамемами** — невозможно (звание — обычное
    существительное, неотличимо от должности), поэтому словарь маркеров.
- **Последствия:** нестандартные заголовки работают без `aliases`; добавлены
  роли звания/личного номера. `recognize()` получил параметр `samples`
  (опциональный — обратная совместимость с прямыми вызовами, существующие
  тесты и header-only путь не трогают pymorphy). `read_table` теперь собирает
  образцы при чтении строк и распознаёт роли **после** (а не до) чтения.
  Контекст-ключ для `position`/`rank`/`personal_number` остаётся **физическим
  заголовком** (в отличие от разбора ФИО с каноническими `Фамилия`/`Имя`/
  `Отчество`) — шаблон ссылается на нормализованный заголовок. Словарь
  `_RANK_MARKERS` компактен (вершинные слова) — пополняется по мере находок;
  неполный словарь молча не опознаёт редкое звание (компенсируется `aliases`).

### 2026-06-22 — Reverse: decline-and-match + run-rewriting + round-trip, без нейро-теггера (T007)

- **Решение:** обратная сборка шаблона (`dyak reverse`, подсистема
  `src/dyak/reverse/`) детерминированная, на **уже имеющемся** стеке
  склонения (petrovich + pymorphy3), без новых тяжёлых зависимостей. Три
  опоры:
  1. **decline-and-match** — кандидат на замену порождает формы по 6
     падежам (`obj.inflect(case)`), и какая форма совпала в тексте, тот
     падежный фильтр и ставится в тег (`{{ Фамилия | дт }}`). Омонимия
     форм (фамилия в рд = вн) → берём первый по каноническому порядку
     (именительный раньше прочих), помечаем `ambiguous` и **подсвечиваем**
     run жёлтым прямо в шаблоне.
  2. **run-rewriting** (`docx_rewrite.py`) — инверсия склейки docxtpl:
     склеиваем run'ы параграфа в плоскую строку + карту границ, находим
     непересекающиеся спаны (длинные раньше), тег целиком кладём в один
     run (владелец начала спана), внутренность спана опустошаем. Так
     forward-docxtpl увидит тег неразрезанным, а форматирование
     литерального текста сохраняется.
  3. **round-trip verify** (`engine.py`) — собранный шаблон тут же
     прогоняется forward-рендером (`render_to_document`, тот же путь, что
     у `generate`) на той же строке, и текст сверяется с образцом;
     расхождение → `roundtrip_mismatch`. Это проверка **верности, не
     полноты**: сопоставленные формы воспроизводятся всегда (matcher и
     forward зовут один и тот же `inflect`), поэтому сверка молчит на
     корректном шаблоне и ловит постороннюю jinja-разметку в образце
     (`{% raw %}`-шпаргалка, забытый тег) или дефект сборки.
- **Контекст:** этап из `CONCEPT.md` §17 — на руках готовый заполненный
  docx, шаблона нет; кадровику нужно «развернуть» образец в шаблон.
  Best-effort: инструмент не претендует на 100%, а честно отчитывается,
  что заменено уверенно (`replaced`), что под сомнением (`ambiguous`),
  что не нашлось (`not_found`) и что похоже на данные без пары
  (`unmatched_text`).
- **Альтернативы:**
  - **Нейро-морфо-теггер `slovnet morph` (Natasha)** для разрешения
    омонимии падежа по контексту — **отложили в T015**, не тащим в T007.
    Причины из Analyze: зонтичный `natasha` на Python 3.14 нерабочий
    (тянет `pymorphy2` → падает на `pkg_resources`); модели ~50 МБ не на
    PyPI (офлайн-bundling, размер Windows-сборки). Главное — на
    **корректность** документов омонимия не влияет (омонимичные падежи
    дают идентичный текст), это улучшение точности тега, а не функции.
  - **Локальный генеративный LLM-fallback** для нечётких мест —
    **отложили в T014** (жёстко локально, опциональный extra).
  - **Перебор подстрок без морфологии** — отвергли: не отличает
    «Иванову» (дт) от «Иванова» (рд/вн), не ставит правильный фильтр.
- **Последствия:** reverse полностью детерминирован и offline,
  0 новых зависимостей; зеркальная forward подсистема переиспользует
  `build_context`/`render_to_document`/`iter_paragraphs` (DRY, без
  дублирования логики ролей/пола/склонения). Точность падежа на омонимии
  ограничена (первый падеж + пометка) — апгрейд в T015 при необходимости.
  Round-trip даёт страховку от регрессий сборки почти бесплатно (один
  forward-прогон на строку-образец).

### 2026-06-22 — PDF-экспорт: прямой вызов `soffice`, не `docx2pdf` (T012)

- **Решение:** `generate --pdf` конвертирует готовые docx в PDF **прямым
  вызовом** `soffice --headless --convert-to pdf` (один батч-запуск на
  весь набор, во временном `-env:UserInstallation`-профиле, чтобы не
  конфликтовать с уже открытым LibreOffice). Бинарник ищется
  кросс-платформенно: `PATH` → типичные пути (Linux `/usr/bin`, `/opt/
  libreoffice*`; macOS `/Applications`; Windows `%ProgramFiles%\
  LibreOffice\program\soffice.exe`). Не найден → доменная
  `PdfExportError` с понятным текстом, не трейсбек.
- **Контекст:** этап 4b (`CONCEPT.md` §12); кадровику иногда нужен PDF.
  Решение Разработчика — кросс-платформенный поиск (Linux + Windows).
- **Альтернативы:** библиотека `docx2pdf` — отвергли: она использует
  **MS Word** (COM на Windows, AppleScript на macOS), а не свободный
  LibreOffice, и не работает headless на Linux-сервере. Конвертировать
  файлы по одному — отвергли (старт LibreOffice дорогой; батч на порядок
  быстрее).
- **Подавление линтера:** `subprocess.run` ловит `S603` (bandit, advisory
  на любой subprocess). Точечный `# noqa: S603` с обоснованием
  согласован с Разработчиком 2026-06-22: argv фиксирован, без `shell`,
  путь из `find_soffice()`, аргументы — пути, сгенерированные приложением;
  пользовательского ввода в argv нет. Глобальный `ignore` не трогали.
- **Последствия:** PDF-экспорт изолирован в `dyak/pdf.py`; внешняя
  зависимость — установленный LibreOffice (не Python-пакет). Реальные
  тесты конвертации skip-guarded (`find_soffice() is None` → skip), логика
  и поиск путей покрыты юнит-тестами с моками — coverage держится и там,
  где LibreOffice нет (CI без него не падает). Windows-пути покрыты
  юнит-тестом с mock'ом env; живой прогон под Windows — при возможности.

### 2026-06-22 — Прогресс generate: rich-бар (TTY) + JSONL в stderr (контракт CLI↔GUI, T013)

- **Решение:** `generate` показывает прогресс двумя независимыми каналами,
  оба в **stderr** (stdout остаётся чист под человеческий итог):
  1. **человекочитаемый бар** через `rich.progress` — рисуется только в
     интерактивном терминале (`sys.stderr.isatty()`); в pipe/файл (в т.ч.
     под GUI) отключён (`disable`) и ничего не печатает;
  2. **машиночитаемый JSONL** по флагу `--progress-json` — построчные
     события, пишутся напрямую в stderr (минуя rich, поэтому корректны и
     когда бар отключён). **Контракт CLI↔GUI** (T008 читает его через
     `subprocess`). Схема событий: `{"event":"start","total":N}` →
     `{"event":"progress","done":k,"total":N,"file":"<имя>"}` (на каждый
     документ) → `{"event":"done","ok":k,"failed":0}`; при прерывании —
     `{"event":"error","done":k,"total":N,"message":"<текст>"}`.
- **Контекст:** пакетные прогоны объёмны (тысячи строк) — без обратной
  связи кажется, что приложение зависло; GUI (T008) — тонкий фронтенд над
  CLI и должен показывать прогресс/ошибки, питаясь машиночитаемым потоком
  (`CONCEPT.md` §15; этап 4 разбит на T005/T012/T013).
- **Альтернативы:** `tqdm` — отвергли (rich богаче, пригодится для
  оформления отчётов; решение Разработчика). Прогресс в stdout — отвергли
  (затрёт результат, который читает скрипт/GUI). Отдельный fd/файл для
  событий — отвергли как избыточный (stderr проще в subprocess-обвязке
  GUI; решение Разработчика). `failed` сейчас всегда 0: `generate` —
  «всё-или-ничего» (первая ошибка прерывает прогон, T004); поле
  зарезервировано на будущую устойчивость per-row (вне scope T013).
- **Последствия:** ядро не знает про вывод — прогресс инкапсулирован в
  `dyak/progress.py` (контекст-менеджер `GenerateProgress`), `generate`
  лишь зовёт `advance(name)` в цикле. Новая runtime-зависимость `rich`.
  Тайминги прогресса не замеряем здесь — это T009. Бинарной завязки на
  TTY в тестах нет: JSONL проверяется на не-TTY, бар — отдельно под `pty`.

### 2026-06-22 — `dyak init`: scaffold-ассеты как bundled package-data (T005)

- **Решение:** `dyak init` выкладывает готовый стартовый набор
  (`dyak.yaml`, `template.docx` со шпаргалкой падежей, `table.xlsx`),

  Шпаргалка падежных фильтров в шаблоне обёрнута в `{% raw %}`, чтобы её
  синтаксис (`{{ ФИО | рд }}`) отображался буквально, а не исполнялся
  при `generate`.
- **Контекст:** этап 4 «Опции» разбит (решение Разработчика) на T005
  (init), T012 (PDF), T013 (прогресс). Нужен способ доставить кадровику
  рабочий пример «из коробки», переживающий и запуск из исходников, и
  будущий бандл (T010).
- **Альтернативы:** генерировать docx/xlsx кодом на лету (`python-docx` /

  того, что проще держать готовым файлом. Копировать из корневого

- **Последствия:** ассеты — бинарники в репозитории (docx/xlsx

  `generate` даёт валидный документ) и тем, что они регенерируемы из
  `python-docx`/`openpyxl` (колонки Фамилия/Имя/Отчество/Должность/Дата
  начала/Дата приказа/Номер приказа; 3 строки: Иванов/Петрова/Ким). В
  T010 сборщик обязан включить `dyak/scaffold/*` в бандл (package-data).

### 2026-06-22 — Надёжность: StrictUndefined глобально, `согл` на ФИО, `check`-эвристики (T004)

- **Решение:**
  1. **`StrictUndefined` — глобально** для всего рендера (тело docx, имя
     файла, `check`). Неизвестная переменная заворачивается в доменную
     `TemplateError` с именем. Молчаливая пустая подстановка опасна для
     кадровых документов — лучше явная ошибка.
  2. **`согл` — на `Declinable`-ФИО** (`{{ ФИО | согл('м','ж') }}`), а не
     на объекте `сотрудник` из CONCEPT §8.3 (его упразднил T006). `Fio`
     несёт пол (`Fio.gender`); применение `согл` к не-ФИО → понятная
     `TemplateError`.
  3. **«Пол-колонка ≠ пол-по-отчеству» переинтерпретирован** под
     отсутствие колонки пола (T006): `check` сверяет ручной `genders` с
     авто-определением (`resolve_gender` с источником). Расхождение и
     дефолт-заглушка → в отчёт.
  4. **`not_declined` — эвристика** (значение не меняется ни в одном
     косвенном падеже). Отчёт advisory: возможны ложные срабатывания на
     легитимно несклоняемых (Дюма, Ким-ж, -ко) — кадровик решает, это
     прямо по CONCEPT §13.
  5. Машиночитаемый отчёт (`--json`) — НЕ здесь; это контракт CLI↔GUI
     (T005/T008). В T004 отчёт человекочитаемый.
- **Контекст:** этап 3 «Надёжность» из `CONCEPT.md` §17; цель — поймать
  проблемы до генерации сотни файлов.
- **Альтернативы:** мягкий `Undefined` (молчит) — отвергли (опасно);
  отдельный объект `сотрудник` ради `согл` — отвергли (T006 ушёл к
  плоскому контексту, ФИО — естественный носитель пола); точный разбор
  «не склонилось» — невозможен без согласования, потому эвристика +
  override.
- **Последствия:** `generate` теперь падает на опечатке в теге (раньше
  молча подставлял пусто) — это осознанное ужесточение. `check` даёт
  кадровику предполётную проверку. Эвристика `not_declined` шумит на
  несклоняемых — компенсируется текстом «если не иностранное слово».

### 2026-06-22 — Должности: пословное склонение, кеш по `(text, case)` без рода, общий анализатор (T003)

- **Решение:**
  1. Должности склоняются **пословно** через `pymorphy3` (`PymorphyInflector`),
     несогласуемые сложные фразы закрываются `overrides.position` с
     приоритетом ДО движка (полный/частичный override по нормализованному
     ключу должности).
  2. **Ключ кеша — `(text, case)` без рода** (отклонение от `CONCEPT.md`
     §7.4, где `(text, case, gender)`): существительные склоняются по
     падежу, род человека на «директора/бухгалтера» не влияет, поэтому
     `PymorphyInflector.inflect(text, case)` без `gender`.
  3. **Один `MorphAnalyzer` на процесс** (`inflection/morph.py`,
     `functools.cache`), общий для движка пола (T002) и должностей —
     `gender.py` переключён на него (убрано дублирование).
- **Контекст:** этап 2 из `CONCEPT.md` §17; `MorphAnalyzer` дорог
  (словарь в мегабайты) — второй экземпляр недопустим.
- **Альтернативы:** держать `gender` в сигнатуре/ключе ради протокольной
  симметрии с `PetrovichInflector` — отвергли (мёртвый параметр, лишний
  множитель в кеше); алгоритмическое согласование прилагательных —
  отвергли (хрупко, R1), вместо него `overrides`.
- **Последствия:** `{{ Должность | рд }}` склоняется; сложные обороты —
  ручной escape-hatch. Кеш эффективен на таблицах с повторяющимися
  должностями. `Inflector`-обёртки ФИО и должностей теперь различаются
  сигнатурой (`gender` есть только у ФИО) — это осознанно, склонение
  прячется за общим портом `Declinable`.

### 2026-06-22 — Склонение в контексте: Declinable-объекты + кириллица через `__getattr__` (T002)

- **Решение:**
  1. Склоняемые значения (`Фамилия`, `Имя`, `Отчество`, `ФИО`) кладутся
     в Jinja-контекст не строками, а доменными объектами `NamePart`/`Fio`
     (`Declinable`, метод `.inflect(case)`). Падежный фильтр вызывает
     `.inflect`, не-`Declinable` значения пропускает как есть. `__str__` =
     именительная форма, поэтому голый `{{ ФИО }}` рендерится нормально.
  2. **Регистр сохраняем в обёртке, а не движком:** `petrovich` не
     склоняет UPPERCASE — для капс-частей склоняем `.capitalize()`-форму
     и поднимаем результат в `.upper()`.
  3. **Кириллические имена инициалов (`{{ ФИО.инициалы }}`) — только в
     DSL шаблона, не в Python-коде.** `Fio.__getattr__` маршрутизирует
     `инициалы`/`инициалы_впереди`/`инициалы_слитно` в объект `Initials`.
     Так ruff `PLC2401` (non-ASCII имена) не нарушается **без** `noqa`.
  4. Пол — автоматически (см. ADR 2026-06-21): отчество → `pymorphy3` по
     имени → ручной `genders` в конфиге → дефолт «мужской» + warning.
- **Контекст:** T006 сделал контекст плоским (заголовок → строка);
  склонению нужна маршрутизация «значение → нужный движок» без английских
  имён в шаблоне и без правки фильтров под каждый новый тип.
- **Альтернативы:**
  - Падежные формы строками в контексте по ключам `Фамилия_рд` — отвергли
    (комбинаторный взрыв, та же причина, что и хранение в данных).
  - Кириллические `@property` на `Fio` — отвергли из-за `PLC2401` и
    жёсткого правила «не глушить линтер»; `__getattr__` чище.
  - `# noqa: PLC2401` / `S701` — отвергли (правило «чинить, а не
    глушить»): `__getattr__` снимает `PLC2401`, `autoescape=True` снимает
    `S701` (см. последствия).
- **Последствия:** новые склоняемые типы (должности T003) подключаются
  через тот же порт `Declinable` без правки фильтров. `Fio` стал
  `Declinable` с `__getattr__` — доступ к произвольным атрибутам, кроме
  инициалов, даёт `AttributeError` (безопасно для dataclass/pickle).
  Имя файла рендерится с `autoescape=True` (как и тело docx): кадровые
  ФИО/номера HTML-спецсимволов не содержат, но экзотический `&`/`'` в
  имени файла экранируется (запарковано как T011).

### 2026-06-21 — Подстановка по заголовкам колонок; пол — автоматически (T006/T002)

Отклонение от DSL/конфига, описанного в `CONCEPT.md` §6, §8–§9. CONCEPT
иммутабелен — фиксируем изменение здесь.

- **Решение:**
  1. Шаблон ссылается на **заголовки колонок напрямую** (`{{ Фамилия }}`,
     `{{ Должность }}`, `{{ Дата_начала }}`), а не через `сотрудник.*` +
     обязательный yaml-маппинг. Стандартные кадровые колонки
     распознаются по названию; `dyak.yaml` опционален (`aliases`,
     `overrides`).
  2. Пробел в заголовке → `_` в теге; забытый пробел чинится
     автоматически + warning (на XML-патче docxtpl).
  3. Имя выходного файла — флаг CLI `--filename` (+ дефолт по ФИО), не в
     обязательном конфиге.
  4. **Колонка «Пол» упраздняется.** Пол определяется автоматически:
     правило по отчеству → fallback pymorphy по имени → `override` +
     сигнал `check`. (`petrovich` сам пол не определяет — поправка к §7.1.)
- **Контекст:** цель проекта — нетехнический пользователь (кадровик).
  yaml-маппинг и ручная колонка пола — барьеры, противоречащие этой цели.
- **Альтернативы:**
  - Оставить `сотрудник.*` + обязательный yaml (как в §8–9) — отвергли
    как сложное для бухгалтера.
  - Тип колонки явным фильтром в шаблоне (`{{ X | фио рд }}`) — отвергли,
    сложнее для пользователя.
  - Колонка «Пол» как обязательная (как в §7, §9, §11) — отвергли в
    пользу автоопределения (исключения редки, закрываются `override`).
- **Последствия:** переделывает `config.py`/контекст из T001 (pre-alpha,
  чистый разрыв формата без миграции данных). Распознавание ролей колонок
  в T006 — фундамент маршрутизации склонения в T002. Реализация: T006
  (подстановка/распознавание) → T002 (склонение + автоопределение пола).
  Детали — `specs/T006-headers-config/spec.md`.

### 2026-06-21 — Запуск через `python -m dyak`, без `[build-system]`

- **Решение:** проект остаётся «application»-style (как шаблон
  dreamteam) — без `[build-system]` и без устанавливаемой console-script.
  Запуск: `PYTHONPATH=src uv run python -m dyak generate …` (точка входа
  `src/dyak/__main__.py` → `cli.app`).
- **Контекст:** `uv` не регистрирует `[project.scripts]` для
  непакетируемого проекта; `CONCEPT.md` §10 показывает UX как `dyak
  generate`.
- **Альтернативы:** добавить `[build-system]` (hatchling) +
  `tool.uv.package = true` ради команды `dyak` и `uv tool install` —
  отвергли на этом этапе (Решение Разработчика), чтобы не менять природу
  проекта; при необходимости дистрибуции пересмотреть отдельной задачей.
- **Последствия:** в документации и примерах команда пишется как
  `python -m dyak …` (а не `dyak …`). Пакетирование/публикация — будущая
  задача, не блокирует функциональность.

### 2026-06-21 — Имя пакета и CLI-команды: `dyak`

- **Решение:** имя дистрибутива, Python-пакета (`src/dyak/`) и CLI —
  `dyak` (транслитерация «дьяк»).
- **Контекст:** `dreamteam init` создал проект в папке `djak`, тогда как
  ТЗ (`CONCEPT.md`) сквозь весь текст использует `dyak`.
- **Альтернативы:** `djak` — отвергли, чтобы не расходиться с ТЗ и
  примерами команд (`dyak generate`).
- **Последствия:** имя корневой папки (`djak`) пока расходится с именем
  пакета — это допустимо (контейнер ≠ дистрибутив). Переименование папки
  `djak → dyak` Разработчик при желании сделает отдельной командой.

### 2026-06-21 — Целевой Python: 3.14

- **Решение:** `requires-python >= 3.14` (планка шаблона dreamteam),
  несмотря на «3.11+» в ТЗ.
- **Контекст:** ТЗ писалось консервативно; основной риск — отсутствие
  колёс под cp314 у морфо-библиотек и `lxml`/`pydantic-core`.
- **Альтернативы:** снизить до 3.11 — не потребовалось.
- **Последствия:** проверено живой установкой и smoke-прогоном
  (`petrovich 2.0.1`, `pymorphy3 2.0.6`, `lxml 6.1.1`, `pydantic-core
  2.46.4`, `docxtpl 0.20.2`): весь стек ставится и склоняет под 3.14.5.
  При проблемах на чужом окружении — планка снижается без правок ядра.

### 2026-06-21 — Падеж кодируется в шаблоне, не в данных; русскоязычный DSL

- **Решение:** целевой падеж — свойство места подстановки, задаётся
  фильтром Jinja с русским сокращением (`им/рд/дт/вн/тв/пр`). Вся
  разметка шаблонов — на русском.
- **Контекст:** в таблице ФИО хранится в именительном; в документе одно
  и то же значение нужно в разных падежах. Пользователи — кадровики, не
  владеют английским.
- **Альтернативы:** хранить падежные формы в данных — отвергли
  (комбинаторный взрыв колонок); английские имена фильтров — отвергли
  (барьер для пользователя).
- **Последствия:** рендер требует адаптера контекста (русский фасад
  `Person`) и регистрации русских фильтров в Jinja-окружении. Подробно —
  `CONCEPT.md` §3, §8.

### 2026-06-21 — Стек: docxtpl + petrovich + pymorphy3, два движка склонения

- **Решение:** рендер docx — `docxtpl` (Jinja2 поверх python-docx);
  склонение ФИО — `petrovich` (по полу); склонение должностей и общих
  слов — `pymorphy3`. Маршрутизация по типу доменного объекта, поверх —
  словарь `overrides` с наивысшим приоритетом.
- **Контекст:** плейсхолдеры docx рвутся по run'ам (прямой find-replace
  ненадёжен); имена собственные и нарицательные требуют разных движков.
- **Альтернативы:** python-docx + ручной find-replace — отвергли
  (хрупко по run'ам); единый морфодвижок для ФИО — отвергли (petrovich
  точнее по полу и несклоняемым фамилиям).
- **Последствия:** склонение прячется за портом `Inflector`/`Declinable`
  (§6 `CONCEPT.md`) — движки заменяемы. Составные должности
  («заместитель генерального директора») автоматика берёт неидеально —
  escape-hatch через `overrides` + отчёт `check`.
