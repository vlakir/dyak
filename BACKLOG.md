# Backlog

Парковка идей, побочных находок и «надо бы потом починить».

**Правило:** если в процессе работы над текущей задачей Claude или
Разработчик замечают что-то постороннее — оно идёт сюда, а не в текущий
коммит. Это защищает от расползания scope.

Это **не формальный таск-трекер** со сроками и метриками — это парковка
идей. Но **порядок имеет значение**: сверху — то, что планируется
ближайшим, ниже — менее срочное (FIFO по умолчанию, можно поднимать
приоритетное наверх). Когда из бэклога что-то берётся в работу — оно
вырастает в задачу или спеку (`specs/T<NNN>-…`) и удаляется отсюда.

## Формат

`- **T<NNN>** — [<дата находки>] <короткое описание> — <опционально: контекст / откуда всплыло>`

ID присваивается при создании; новый = `max(существующих T-ID в
BACKLOG.md, BOARD.md и CHANGELOG.md) + 1`. ID не переиспользуется
и сохраняется при перетекании задачи между BACKLOG и BOARD; после
релиза задача переходит в `CHANGELOG.md` (с тем же T-ID), что
гарантирует уникальность между релизами.

- **T011** — [2026-06-22] Имена файлов с HTML-спецсимволами (`& < > " '`)
  экранируются. В T002 имя файла рендерится единым Jinja-env с
  `autoescape=True` (нужно для XML тела docx; ruff `S701`), поэтому
  редкий `&`/`'` в ФИО/значении уедет в `&amp;`/`&#39;` в имени файла.
  Для кадровых русских ФИО/номеров не встречается, но для полноты —
  починить (отдельный рендер имени без HTML-экранирования или
  пост-unescape). Низкий приоритет. Откуда всплыло: ADR 2026-06-22 (T002).
  Acceptance: имя файла из значения с `&`/`'` сохраняется буквально;
  тело docx по-прежнему корректно экранируется.

> **Этап 4 «Опции» разбит** (решение Владимира 2026-06-22) на три

- **T012** — [2026-06-22] Этап 4b «Опции: PDF». Опциональный экспорт
  результата в PDF через LibreOffice headless: `--pdf` конвертирует
  каждый сгенерированный docx (`soffice --headless --convert-to pdf`).
  **Кросс-платформенный поиск бинарника** (решение Владимира): PATH +
  типичные пути установки (Linux: `/usr/bin/soffice`, `/opt/libreoffice*/
  program/soffice`; Windows: `%ProgramFiles%\LibreOffice\program\
  soffice.exe`, `%ProgramFiles(x86)%\...`). LibreOffice не найден →
  внятное сообщение + ненулевой код/skip, **не трейсбек**. Выбор подхода
  (прямой вызов `soffice` vs `docx2pdf`) — ADR при взятии. (`CONCEPT.md`
  §12)
  Acceptance: `--pdf` кладёт pdf рядом с docx; `soffice` ищется в PATH и
  стандартных директориях на Linux **и** Windows; отсутствие LibreOffice →
  понятное сообщение + ненулевой код/skip без трейсбека; pdf соответствует
  docx. Тестирование на Windows затруднено в текущем окружении — путевой
  поиск покрыть юнит-тестом с mock'ом ОС/файловой системы.

- **T013** — [2026-06-22] Этап 4c «Опции: прогресс + контракт CLI↔GUI».
  Живой индикатор прогресса при `generate` через **`rich.progress`**
  (решение Владимира): счётчик `N/Total` + бар; авто-detect — в
  неинтерактивном выводе (pipe/файл, не TTY) бар не печатается.
  Параллельно — **машиночитаемый прогресс по флагу `--progress-json`**
  (решение Владимира): построчные JSON-события в **stderr**
  (`{"event":"progress","done":k,"total":N}` … финальное
  `{"event":"done","ok":..,"failed":..}`), **stdout остаётся чист** под
  результат. Контракт потребляет GUI (T008) через `subprocess`. Прогресс
  не должен искажать тайминги (замер — T009). (`CONCEPT.md` §15)
  Acceptance: при `generate` на тысячи строк в TTY виден живой бар
  (`N/Total`); в pipe/файл бар не засоряет вывод; `--progress-json`
  эмитит валидный JSONL в stderr (событие на документ + финальное
  `done`), парсится построчно; stdout не содержит прогресс-шум;
  человекочитаемый бар и `--progress-json` могут работать вместе.

- **T007** — [2026-06-22] Авто-генерация шаблона из готового документа
  (reverse-инструмент). Пользователь делает обычный заполненный документ
  для одной строки таблицы (без разметки), `dyak` сопоставляет его со
  строкой и выдаёт docx-шаблон с jinja-тегами `{{ Заголовок }}`
  (+ падежные фильтры). Команда вроде `dyak scaffold-template --doc … --table
  … --row N --out template.docx`.
  **Зависит от T002/T003** — для распознавания склонённых форм нужны
  движки склонения (иначе матчатся только именительные значения).
  Подход — **ассистент-черновик, не «магия»**: генерим шаблон + отчёт
  (что заменено уверенно / неоднозначно / не найдено), кадровик
  проверяет. Сложности зафиксировать в спеке при взятии: склонённые
  формы (склоняем кандидата по 6 падежам и матчим → нужный фильтр);
  неоднозначные/повторяющиеся/короткие значения; пустые ячейки; даты,
  совпавшие случайно; регистр (UPPERCASE фамилия); поиск по склеенному
  тексту параграфа с переписыванием run'ов (run-split). MVP (без
  склонения, только точные именительные совпадения) можно сделать
  раньше как отдельную фазу.
  **Несопоставимые данные → не падаем, а предупреждаем.** Если в
  примере-документе есть значения, которых нет в строке таблицы
  (пользователь вписал что-то «от себя» / опечатался / поле не из
  таблицы), приложение НЕ прерывается: для каждого такого фрагмента —
  warning в лог/отчёт, а сам фрагмент остаётся в шаблоне как статический
  текст (константа). Цель — всё равно «вытянуть» логику и выдать рабочий
  шаблон (best-effort), а разбор несопоставленных мест оставить кадровику
  через отчёт.
  Acceptance (черновой, уточнить в `specs/T007-*/spec.md`): на эталонном
  заполненном документе + строке таблицы генерится валидный шаблон, по
  которому `generate` воспроизводит исходный документ; именительные
  значения (даты/номера/должность) заменяются на теги; склонённые ФИО
  заменяются на тег с правильным падежным фильтром (после T002);
  неоднозначные/ненайденные места попадают в отчёт, а не молча
  пропускаются/портятся; **значения из документа, которых нет в таблице,
  не валят процесс** — warning + остаются статическим текстом, шаблон всё
  равно генерится (best-effort); крупная фича — идём через
  spec/clarify/analyze.

  **Design-note (снижение % ошибок, ТОЛЬКО локально — данные не утекают).**
  Жёсткое требование Разработчика: никакого облака, обработка на машине
  пользователя. Архитектура **deterministic-first**:
  1. Основа — локальный русский морфо-стек (без генеративного LLM):
     `pymorphy3` (словарный морфоанализ, падеж слова; уже в стеке для
     T002/T003) + `Natasha`/`Slovnet morph` (контекстный нейро-теггер
     падежа/рода/числа, модели в мегабайты, CPU). Определение падежа спана
     и выбор фильтра решается этим, без LLM — меньше железа, детерминизм,
     приватность.
  2. Опционально, второй фазой — **локальный** генеративный LLM как
     fallback для нечётких мест (перефразировки, редкие формы): небольшие
     русские open-weight модели `T-lite`/`T-pro 2.0` (Т-Банк),
     `Vikhr` (Mistral-7B), `Saiga` (Llama-3 8B / Mistral); запуск через
     Ollama/llama.cpp/LM Studio, 7–8B в квантовании на GPU или CPU. Всё
     локально — ничего наружу.
  3. Страховка независимо от источника замены — **round-trip verify**:
     рендерим сгенерированный шаблон обратно через `generate` и сравниваем
     с исходным документом; не совпало → в отчёт, не в тихую порчу.
  Опциональную ИИ-зависимость оформить отдельным extra/модулем (без
  запрещённого `try/except import`); решение по стеку — ADR при взятии.

- **T008** — [2026-06-22] GUI для пользователя (финал, «вишенка на торте»).
  Графическая оболочка в привычном оконном окружении: выбор шаблона
  (docx), таблицы (xlsx), папки назначения и всех значимых опций (имя
  файла, лист, конфиг; позже — `--pdf`, локальный AI-режим T007). Кнопка
  «Сгенерировать» → прогресс → понятный результат/ошибка в окне. Внутри —
  **встроенный подробный help** (оффлайн, по разделам, для нетехнического
  пользователя).
  **Прогресс на больших объёмах:** живой индикатор (прогресс-бар +
  счётчик `N/Total`), запитанный машиночитаемым прогрессом из CLI (см.
  T005 / контракт CLI↔GUI). Цель — при пакетах в несколько тысяч строк
  пользователь видит движение и уверен, что приложение не зависло; окно
  не должно «замерзать» (генерация в фоне, UI отзывчив).
  **Архитектура — тонкий фронтенд над CLI:** GUI не дублирует логику, а
  вызывает `dyak` через `subprocess`; ядро остаётся единственным
  источником истины. Следствие: CLI, вероятно, нужно дооснастить
  машиночитаемым выводом (`--json`/коды возврата/прогресс), чтобы GUI
  показывал результат и ошибки внятно — зафиксировать в спеке.
  **Стек (на усмотрение, финал — ADR при взятии):** рекомендация —
  `PySide6` (Qt) ради нативного вида, нормальных файловых диалогов и
  богатого виджета для help; лёгкая альтернатива без зависимостей —
  `tkinter` (stdlib), компромисс — `customtkinter`. Кросс-платформенность:
  Linux обязательно, Windows желательно.
  **Зависит от стабильного CLI** — берётся последней, после core-этапов
  (T002–T005) и желательно T007. Крупная фича → spec/clarify/analyze + ADR
  (выбор GUI-стека и контракт CLI↔GUI).
  Acceptance (черновой, уточнить в `specs/T008-*/spec.md`): окно с
  выборщиками шаблона/таблицы/папки и значимых опций; запуск генерации
  через `subprocess` к `dyak` (логика не продублирована в GUI); прогресс
  и человекочитаемый результат/ошибка в окне (не «молчит» и не падает
  трейсбеком); встроенный подробный оффлайн-help по разделам; запускается
  на Linux в оконном окружении; рассчитан на нетехнического пользователя
  (кадровик доходит от «открыл» до «получил документы» без терминала);
  при генерации тысяч строк виден живой прогресс, окно не «замерзает».

- **T009** — [2026-06-22] Финал: нагрузочное тестирование + замер
  производительности + выявление узких мест. Закрывающий этап — после
  core (T002–T005), T007 и GUI (T008), когда система собрана целиком.
  На объёмных батчах (несколько тысяч строк): прогнать `generate`,
  замерить время и память, профилировать (`cProfile`/`pyinstrument`),
  найти хотспоты (чтение xlsx, склонение ФИО/должностей, рендер docx,
  запись файлов) и устранить либо запарковать с обоснованием. Зафиксировать
  целевые показатели производительности (taргет для T005/T008 прогресса).
  Acceptance (черновой, оформить spec/plan при взятии): воспроизводимый
  нагрузочный сценарий на N тысяч строк (генератор фикстуры); зафиксированы
  замеры времени/памяти; профиль выявляет узкие места; критичные узкие
  места устранены или запаркованы с обоснованием; индикатор прогресса
  (CLI и GUI) корректно работает на больших объёмах и не искажает тайминги;
  целевой показатель (тысячи строк за <целевое время>) достигнут либо
  отклонение зафиксировано с причиной.

- **T010** — [2026-06-22] Сборка Windows-бинарников через GitHub Actions
  (после GUI T008). На каждый релиз — две версии: **setup-инсталлятор** и
  **portable**. По образцу `mil_mapper` (там:
  `.github/workflows/build-windows-installer.yml` + `SK42.spec` /
  `SK42_portable.spec` + `installer.iss`):
  - **Workflow** на `windows-latest`, триггер по тегу (`v*` / `[0-9]*`) +
    `workflow_dispatch`; `VERSION` из имени тега (`github.ref_name`).
  - **Сборка GUI** (T008) через **PyInstaller**: onedir-спека для
    инсталлятора + отдельная спека для portable.
  - **Setup** — Inno Setup (`iscc installer.iss`), per-user установка без
    админ-прав (`{localappdata}`), RU/EN, иконка → `dyak-<version>-setup.exe`.
  - **Portable** — собрать папку (exe + README + место под конфиг/вывод
    рядом с программой), заархивировать → `dyak-<version>-portable.zip`.
  - **Release** — `softprops/action-gh-release@v2` на тег, приложить оба
    артефакта.
  **Отличия от mil_mapper под наш стек:** mil_mapper на Poetry — у нас
  **uv** (`uv sync`, `uv run pyinstaller`, кеш `.venv` по `uv.lock`);
  сверить доступность PyInstaller-колёс под целевой Python (mil_mapper на
  3.13; у нас 3.14 — при проблемах зафиксировать build-Python отдельно);
  GUI на PySide6/Qt (T008) → проверить включение Qt-плагинов в бандл и его
  размер. ADR T006 (без `[build-system]`) не мешает: PyInstaller собирает
  из entry-скрипта.
  **Зависит от T008** (нужен собранный GUI-вход). Инфра-задача с T-ID;
  ADR при взятии (PyInstaller + Inno Setup, uv-адаптация workflow).
  Acceptance (черновой): пуш тега `vX.Y.Z` запускает workflow на
  `windows-latest`; PyInstaller собирает GUI; Inno Setup даёт
  `dyak-X.Y.Z-setup.exe` (per-user, без админа); portable-zip
  `dyak-X.Y.Z-portable.zip` (распаковал и запустил без установки); оба
  артефакта прикреплены к GitHub Release; сборка воспроизводима из чистого
  клона.
