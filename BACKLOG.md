# Backlog

Парковка идей, побочных находок и «надо бы потом починить».

**Правило:** если в процессе работы над текущей задачей Claude или
Разработчик замечают что-то постороннее — оно идёт сюда, а не в текущий
коммит. Это защищает от расползания scope.

Это **не формальный таск-трекер** со сроками и метриками — это парковка
идей. Но **порядок имеет значение**: сверху — то, что планируется
ближайшим, ниже — менее срочное (FIFO по умолчанию, можно поднимать
приоритетное наверх). Когда из бэклога что-то берётся в работу — оно
вырастает в задачу или спеку (`specs/T<NNN>-…`) и удаляется отсюда.

## Формат

`- **T<NNN>** — [<дата находки>] <короткое описание> — <опционально: контекст / откуда всплыло>`

ID присваивается при создании; новый = `max(существующих T-ID в
BACKLOG.md, BOARD.md и CHANGELOG.md) + 1`. ID не переиспользуется
и сохраняется при перетекании задачи между BACKLOG и BOARD; после
релиза задача переходит в `CHANGELOG.md` (с тем же T-ID), что
гарантирует уникальность между релизами.

- **T007** — [2026-06-22] Авто-генерация шаблона из готового документа
  (reverse-инструмент). Пользователь делает обычный заполненный документ
  для одной строки таблицы (без разметки), `dyak` сопоставляет его со
  строкой и выдаёт docx-шаблон с jinja-тегами `{{ Заголовок }}`
  (+ падежные фильтры). Команда вроде `dyak scaffold-template --doc … --table
  … --row N --out template.docx`.
  **Зависит от T002/T003** — для распознавания склонённых форм нужны
  движки склонения (иначе матчатся только именительные значения).
  Подход — **ассистент-черновик, не «магия»**: генерим шаблон + отчёт
  (что заменено уверенно / неоднозначно / не найдено), кадровик
  проверяет. Сложности зафиксировать в спеке при взятии: склонённые
  формы (склоняем кандидата по 6 падежам и матчим → нужный фильтр);
  неоднозначные/повторяющиеся/короткие значения; пустые ячейки; даты,
  совпавшие случайно; регистр (UPPERCASE фамилия); поиск по склеенному
  тексту параграфа с переписыванием run'ов (run-split). MVP (без
  склонения, только точные именительные совпадения) можно сделать
  раньше как отдельную фазу.
  Acceptance (черновой, уточнить в `specs/T007-*/spec.md`): на эталонном
  заполненном документе + строке таблицы генерится валидный шаблон, по
  которому `generate` воспроизводит исходный документ; именительные
  значения (даты/номера/должность) заменяются на теги; склонённые ФИО
  заменяются на тег с правильным падежным фильтром (после T002);
  неоднозначные/ненайденные места попадают в отчёт, а не молча
  пропускаются/портятся; крупная фича — идём через spec/clarify/analyze.

  **Design-note (снижение % ошибок, ТОЛЬКО локально — данные не утекают).**
  Жёсткое требование Разработчика: никакого облака, обработка на машине
  пользователя. Архитектура **deterministic-first**:
  1. Основа — локальный русский морфо-стек (без генеративного LLM):
     `pymorphy3` (словарный морфоанализ, падеж слова; уже в стеке для
     T002/T003) + `Natasha`/`Slovnet morph` (контекстный нейро-теггер
     падежа/рода/числа, модели в мегабайты, CPU). Определение падежа спана
     и выбор фильтра решается этим, без LLM — меньше железа, детерминизм,
     приватность.
  2. Опционально, второй фазой — **локальный** генеративный LLM как
     fallback для нечётких мест (перефразировки, редкие формы): небольшие
     русские open-weight модели `T-lite`/`T-pro 2.0` (Т-Банк),
     `Vikhr` (Mistral-7B), `Saiga` (Llama-3 8B / Mistral); запуск через
     Ollama/llama.cpp/LM Studio, 7–8B в квантовании на GPU или CPU. Всё
     локально — ничего наружу.
  3. Страховка независимо от источника замены — **round-trip verify**:
     рендерим сгенерированный шаблон обратно через `generate` и сравниваем
     с исходным документом; не совпало → в отчёт, не в тихую порчу.
  Опциональную ИИ-зависимость оформить отдельным extra/модулем (без
  запрещённого `try/except import`); решение по стеку — ADR при взятии.

- **T008** — [2026-06-22] GUI для пользователя (финал, «вишенка на торте»).
  Графическая оболочка в привычном оконном окружении: выбор шаблона
  (docx), таблицы (xlsx), папки назначения и всех значимых опций (имя
  файла, лист, конфиг; позже — `--pdf`, локальный AI-режим T007). Кнопка
  «Сгенерировать» → прогресс → понятный результат/ошибка в окне. Внутри —
  **встроенный подробный help** (оффлайн, по разделам, для нетехнического
  пользователя).
  **Архитектура — тонкий фронтенд над CLI:** GUI не дублирует логику, а
  вызывает `dyak` через `subprocess`; ядро остаётся единственным
  источником истины. Следствие: CLI, вероятно, нужно дооснастить
  машиночитаемым выводом (`--json`/коды возврата/прогресс), чтобы GUI
  показывал результат и ошибки внятно — зафиксировать в спеке.
  **Стек (на усмотрение, финал — ADR при взятии):** рекомендация —
  `PySide6` (Qt) ради нативного вида, нормальных файловых диалогов и
  богатого виджета для help; лёгкая альтернатива без зависимостей —
  `tkinter` (stdlib), компромисс — `customtkinter`. Кросс-платформенность:
  Linux обязательно, Windows желательно.
  **Зависит от стабильного CLI** — берётся последней, после core-этапов
  (T002–T005) и желательно T007. Крупная фича → spec/clarify/analyze + ADR
  (выбор GUI-стека и контракт CLI↔GUI).
  Acceptance (черновой, уточнить в `specs/T008-*/spec.md`): окно с
  выборщиками шаблона/таблицы/папки и значимых опций; запуск генерации
  через `subprocess` к `dyak` (логика не продублирована в GUI); прогресс
  и человекочитаемый результат/ошибка в окне (не «молчит» и не падает
  трейсбеком); встроенный подробный оффлайн-help по разделам; запускается
  на Linux в оконном окружении; рассчитан на нетехнического пользователя
  (кадровик доходит от «открыл» до «получил документы» без терминала).
