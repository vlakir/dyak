# Пример запуска dyak

Готовый набор для ручного прогона генерации документов.

| Файл | Назначение |
|---|---|
| `employees.xlsx` | Таблица сотрудников (3 строки), стандартные заголовки, без колонки «Пол». |
| `employees_nonstandard.xlsx` | Та же таблица, но с **нестандартным заголовком** «Сотрудник» (одна колонка ФИО вместо трёх) — распознаётся **по содержимому** без `aliases` (T016). |
| `personnel_ranks.xlsx` + `rank_order_template.docx` | Госструктура: колонки «Звание» и «Личный номер». Звание **склоняется** (составное — голова склоняется, хвост замирает), личный номер подставляется буквально (T016 фаза B). |
| `order_template.docx` | Шаблон приказа со **склонением ФИО и должности** по падежам: `{{ ФИО \| вн }}`, `{{ Должность \| рд }}`, инициалы `{{ ФИО.инициалы_впереди }}`, плюс простые теги по заголовкам (`{{ Дата_начала }}`). |
| `military_units.xlsx` + `military_order_template.docx` | **Склонение по умолчанию** (T024): «Подразделение» склоняется **без объявления роли** — `{{ Подразделение \| вн }}` → «роту связи», «взвод охраны»; неодуш. «1 мотострелковый батальон» в вин. = им.; составная должность через дефис «механик-водитель» → «механику-водителю». Заодно колонка **«л/н»** со спецсимволом — тег `{{ л/н }}` работает (T026). Результаты — в `military_output/`. |
| `dyak.yaml` | **Опциональный** конфиг (aliases / overrides). Для этого примера не нужен. |

## Старт с нуля: `dyak init` (T005)

Если своего набора ещё нет — `dyak init` положит готовый стартовый
комплект (тот же по смыслу, что файлы рядом), с которого удобно начать:

```bash
PYTHONPATH=src uv run python -m dyak init --dir ./my_project
```

Создаст в каталоге три файла:

- `dyak.yaml` — опциональный конфиг (всё закомментировано);
- `template.docx` — пример приказа со **шпаргалкой падежных фильтров**
  вверху (блок «ШПАРГАЛКА» удаляется перед использованием);
- `table.xlsx` — пример таблицы сотрудников (3 строки).

Сразу после этого набор прогоняется генерацией:

```bash
PYTHONPATH=src uv run python -m dyak generate \
  --table ./my_project/table.xlsx \
  --template ./my_project/template.docx \
  --out ./my_project/output
```

Повторный `init` не затирает существующие файлы — для перезаписи нужен
`--force`.

## Запуск (нулевой конфиг)

Из корня проекта:

```bash
uv sync   # один раз, если ещё не делали

PYTHONPATH=src uv run python -m dyak generate \
  --table examples/employees.xlsx \
  --template examples/order_template.docx \
  --out examples/output \
  --filename "Приказ_{{ Номер_приказа }}_{{ Фамилия }}.docx"
```

Результат — три файла в `examples/output/`:

```
Приказ_17_Иванов.docx
Приказ_18_Петрова.docx
Приказ_19_Ким.docx
```

Внутри — корректно **просклонённые** ФИО и должности (пол определён
автоматически, колонка «Пол» не нужна):

```
Назначить Иванова Петра Семёновича на должность директора ...
Основание: личное заявление Иванова Петра Семёновича.
П. С. Иванов с приказом ознакомлен: ...

Назначить Петрову Анну Сергеевну на должность главного бухгалтера ...
А. С. Петрова с приказом ознакомлена: ...

Назначить Кима Олега Викторовича на должность инженера-программиста ...
```

Обрати внимание: «ознакомлен» для мужчин и «ознакомлена» для женщины —
это фильтр согласования по полу `{{ ФИО | согл('ознакомлен',
'ознакомлена') }}` (T004).

Конфиг (`--config`) не передаётся — подстановка идёт напрямую по
заголовкам колонок таблицы. Имя файла задаётся флагом `--filename`
(тот же Jinja-движок). Если `--filename` опустить, имя соберётся по
распознанным колонкам ФИО: `Иванов_Пётр_Семёнович.docx`.

## Прогресс на больших объёмах (T013)

В интерактивном терминале `generate` сам показывает живой прогресс-бар
(`N/Total` + ETA) — на пакете в тысячи строк видно, что приложение не
зависло. В pipe/файл бар не печатается (не мусорит в лог).

Для интеграции (будущий GUI, T008) есть машиночитаемый поток — флаг
`--progress-json` эмитит построчный JSONL в **stderr**, а stdout
остаётся чистым под результат:

```bash
PYTHONPATH=src uv run python -m dyak generate \
  --table examples/employees.xlsx \
  --template examples/order_template.docx \
  --out examples/output \
  --progress-json 2>progress.jsonl
```

В `progress.jsonl` — события по одному на строку:

```json
{"event": "start", "total": 3}
{"event": "progress", "done": 1, "total": 3, "file": "Иванов_Пётр_Семёнович.docx"}
{"event": "done", "ok": 3, "failed": 0}
```

## PDF-экспорт (T012)

Если нужен ещё и PDF — добавь флаг `--pdf`: рядом с каждым docx ляжет
PDF, сконвертированный через LibreOffice (`soffice --headless`):

```bash
PYTHONPATH=src uv run python -m dyak generate \
  --table examples/employees.xlsx \
  --template examples/order_template.docx \
  --out examples/output \
  --pdf
```

Нужен установленный **LibreOffice** (Linux/macOS/Windows; ищется в PATH и
типичных путях установки). Если его нет — `generate --pdf` завершится
понятной ошибкой и кодом 1 (docx при этом успеют сгенерироваться), а не
трейсбеком.

## Сухой прогон перед генерацией

Перед генерацией сотен файлов прогони проверку — она рендерит всё в
память (ничего не пишет) и печатает отчёт о подозрительных местах
(не просклонившиеся слова, неуверенно определённый пол, опечатки в
тегах шаблона):

```bash
PYTHONPATH=src uv run python -m dyak check \
  --table examples/employees.xlsx \
  --template examples/order_template.docx
```

На этом примере отчёт — «Проблем не найдено» (код возврата 0). Если в
шаблоне опечататься в теге (`{{ Должнсть }}`), `check` назовёт переменную
и завершится кодом 1 — то же самое прервёт и `generate`, поэтому удобно
ловить такое заранее.

## Обратная сборка шаблона из готового документа (T007)

Бывает наоборот: на руках уже **заполненный** docx (образец приказа),
а шаблона с тегами нет. `dyak reverse` собирает шаблон из такого образца
и строки таблицы, по которой он сделан, — находит в тексте значения
строки (в том числе **в косвенных падежах** и инициалы) и подставляет
вместо них jinja-теги с нужными падежными фильтрами.

Сначала получим образец — сгенерируем один документ обычным `generate`
(в реальной жизни образец приносит кадровик):

```bash
PYTHONPATH=src uv run python -m dyak generate \
  --table examples/employees.xlsx \
  --template examples/order_template.docx \
  --out examples/output
```

Теперь соберём из первого документа шаблон (строка 1 — тот же Иванов):

```bash
PYTHONPATH=src uv run python -m dyak reverse \
  --doc "examples/output/Иванов_Пётр_Семёнович.docx" \
  --table examples/employees.xlsx \
  --row 1 \
  --out examples/output/reversed_template.docx
```

В отчёте видно, что заменено уверенно, а что — с оговоркой по падежу
(омонимия форм, напр. фамилия в родительном и винительном совпадает):

```
Заменено значений: 4. Прочих замечаний: 4.
Заменено:
  Номер_приказа: «17» → {{ Номер_приказа }}
  Дата_приказа: «30.06.2026» → {{ Дата_приказа }}
  Дата_начала: «01.07.2026» → {{ Дата_начала }}
  ФИО.инициалы_впереди: «П. С. Иванов» → {{ ФИО.инициалы_впереди }}
Под сомнением:
  Фамилия: «Иванов» → {{ Фамилия | рд }} (падеж под вопросом) (×2)
  Имя: «Пётр» → {{ Имя | рд }} (падеж под вопросом) (×2)
  Отчество: «Семёнович» → {{ Отчество | рд }} (падеж под вопросом) (×2)
  Должность: «директор» → {{ Должность | рд }} (падеж под вопросом)
```

Места с неоднозначным падежом дополнительно **подсвечены жёлтым** прямо
в собранном шаблоне — кадровику видно, что проверить руками. Инструмент
**best-effort**: он не гарантирует 100%-ный шаблон, а честно показывает,
что заменил, а что осталось статикой (`Не найдено в документе`,
`Похоже на данные без пары в строке`).

Под капотом reverse делает **обратную сверку** (round-trip): собранный
шаблон тут же прогоняется через `generate` на той же строке, и результат
сравнивается с образцом. Если шаблон не воспроизводит исходный документ —
это попадёт в отчёт строкой `Расхождение при обратной сверке` (на чистом
образце таких строк нет).

**Подпись и блок исполнителя (T018).** Reverse ищет в документе **только
значения указанной строки** — он не размечает «по своему разумению» и чужую
фамилию (подписант, исполнитель) не трогает. Но если подпись/исполнитель —
**тот же человек**, что субъект (та же фамилия, или документ самоподписанный),
его фамилия/инициалы в подписи тоже получат тег, хотя подпись обычно
постоянная. На этот случай reverse, когда имя субъекта заменено **в
нескольких местах**, печатает предупреждение:

```
Проверьте подпись/исполнителя:
  имя субъекта заменено тегами в 2 местах — проверьте, нет ли среди них
  подписи или блока исполнителя (там фамилия обычно постоянная; уберите тег,
  оставив текст)
```

Кадровик смотрит шаблон и при необходимости убирает тег из подписи, оставив
постоянное имя текстом.

**Звания в reverse (T017).** Колонка «Звание» тоже разворачивается в
падежный тег — на образце из `personnel_ranks.xlsx`:

```bash
PYTHONPATH=src uv run python -m dyak generate \
  --table examples/personnel_ranks.xlsx \
  --template examples/rank_order_template.docx \
  --out examples/output --filename "{{ Фамилия }}.docx"

PYTHONPATH=src uv run python -m dyak reverse \
  --doc "examples/output/Сидоров.docx" \
  --table examples/personnel_ranks.xlsx --row 1 \
  --out examples/output/reversed_rank.docx
```

В отчёте «майор медицинской службы» (просклонённое в образце как «майору
медицинской службы») разворачивается в `{{ Звание | дт }}` — голова в
дательном, генитивный хвост в теге сохраняется; «Личный номер» (несклоняемый)
становится буквальным `{{ Личный_номер }}`. Падежная омонимия головы звания
(рд = вн) помечается как «под вопросом» — так же, как у ФИО и должностей.

## Распознавание колонок по содержимому (T016)

Заголовки таблицы редко стандартны: «Сотрудник» вместо «Фамилия», «ФИО»,
«Наименование должности». Раньше нестандартный заголовок приходилось
привязывать к роли вручную через `aliases` в `dyak.yaml`. Теперь dyak
**смотрит на содержимое ячеек** (через тот же `pymorphy3`, без новых
зависимостей) и определяет роль сам — `aliases` остаётся ручным override
с высшим приоритетом, но для типичных таблиц не нужен.

`employees_nonstandard.xlsx` — та же тройка сотрудников, но ФИО в **одной**
колонке с заголовком «Сотрудник» (не «ФИО»). Прогон **без** `--config`
(без `aliases`!):

```bash
PYTHONPATH=src uv run python -m dyak generate \
  --table examples/employees_nonstandard.xlsx \
  --template examples/order_template.docx \
  --out examples/output
```

dyak видит, что в «Сотрудник» лежат полные ФИО (грамемы фамилия+имя+
отчество), распознаёт колонку как ФИО, разбирает на части и склоняет —
тот же результат, что и со стандартной таблицей:

```
Назначить Иванова Петра Семёновича на должность директора с 01.07.2026.
П. С. Иванов с приказом ознакомлен: ...
```

Как это работает по приоритету:

1. **`aliases`** из `dyak.yaml` — если задан, главнее всего.
2. **Заголовок-синоним** — «Фамилия», «Должность», «Звание» и т.п.
   распознаются по названию.
3. **Содержимое** — когда заголовок незнаком, роль определяется по
   образцам ячеек: фамилия/имя/отчество (или целое ФИО), должность,
   **звание** (по словарю званий), **личный номер** (код из цифр), дата.
   Если содержимое **единогласно** противоречит знакомому заголовку
   (заголовок «Имя», а в ячейках явные должности) — dyak берёт содержимое
   и предупреждает (ловит ошибку разметки). При неуверенном распознавании
   берётся лучшая догадка с предупреждением.

## Склонение воинских/служебных званий (T016, фаза B)

Кадровики госструктур (армия, флот, МВД, медслужба, юстиция) работают с
колонками **«Звание»** и **«Личный номер»**. Звание склоняется не как
должность: у составного звания склоняется только **голова**, а
несогласованный хвост в родительном замирает.

`personnel_ranks.xlsx` + `rank_order_template.docx` — пример приказа с
благодарностью. Прогон:

```bash
PYTHONPATH=src uv run python -m dyak generate \
  --table examples/personnel_ranks.xlsx \
  --template examples/rank_order_template.docx \
  --out examples/output --filename "{{ Фамилия }}.docx"
```

Шаблон: `{{ Звание | дт }} {{ ФИО | дт }} (личный номер {{ Личный_номер }})`.
Результат:

```
майору медицинской службы Сидорову Ивану Петровичу (личный номер А-123456)
капитану 3 ранга Кузнецову Олегу Викторовичу (личный номер Б-654321)
контр-адмиралу Волкову Павлу Сергеевичу (личный номер В-789012)
```

Видно: **«майору медицинской службы»** (а не «…службе»), **«капитану 3
ранга»** (а не «…рангу»), дефисное **«контр-адмиралу»** (приставка не
склоняется), имя собственное в хвосте сохраняет регистр; «Личный номер» —
буквально. Колонки распознаны без `aliases`: «Звание» — по заголовку,
«Личный номер» — по заголовку, «Сотрудник» — по содержимому.

Редкое звание, которое движок берёт неидеально, закрывается вручную —
секцией `overrides.rank` в `dyak.yaml` (по аналогии с `overrides.position`):

```yaml
overrides:
  rank:
    "генерал-полковник авиации":
      дт: "генерал-полковнику авиации"
```

## Пустые опциональные ячейки: чистый пропуск (T016, фаза C)

Часть полей опциональна — у кого-то нет отчества, звания, премии. Если в
таблице ячейка пустая, в готовом документе на её месте будет **чистый
пропуск**, а не двойной пробел, «пробел перед точкой» или висячая
пунктуация. Это работает **автоматически** при любом `generate` — отдельной
опции не нужно.

Что dyak убирает при пустом значении (на примере шаблонных фраз):

| В шаблоне (значение пустое) | В документе |
|---|---|
| `Назначить {{ Звание }} {{ Фамилия }}.` | `Назначить Петров.` |
| `Оклад {{ Премия }}, надбавка.` | `Оклад, надбавка.` |
| `Звание: {{ Звание }}.` | `Звание:` |
| `{{ Звание }} Петров назначен.` | `Петров назначен.` |

Видно: сдвоенный пробел схлопывается, пробел перед запятой/точкой убирается,
ведущий пробел подрезается, а **висячая точка** в конце строки при пустом
значении убирается целиком (запятая-разделитель — сохраняется). Тире,
открывающие кавычки/скобки и непустые значения не затрагиваются. Если же при
пустом значении должна исчезать **вся строка** — это уже условная разметка
(`{% if Звание %}…{% endif %}`), которую docxtpl поддерживает, но для
нетехнического пользователя она вне основного сценария.

## Фильтр в таблице: генерация только по видимым строкам (T019)

Если в таблице включён **автофильтр** (или часть строк скрыта вручную), dyak
формирует документы **только из видимых** строк — отфильтрованные пропускает
(в логе: «Пропущено скрытых (отфильтрованных) строк: N»). Это удобно, когда
из большого списка нужно сгенерировать пакет только по выбранным сотрудникам:
отфильтровал в Excel, сохранил, запустил `generate`.

При этом **распознавание ролей колонок** (см. «Распознавание колонок по
содержимому») использует **все** строки, включая скрытые: фильтр сужает
выборку документов, но не мешает dyak правильно определить, где фамилия, где
должность. Например, можно отфильтровать таблицу до одной строки — роли всё
равно определятся по всему содержимому.

## Что попробовать

- **Падежи ФИО.** Поменяй падежный фильтр у `{{ ФИО | … }}`: `им` (кто?),
  `рд` (кого?), `дт` (кому?), `вн` (кого? — винительный), `тв` (кем?),
  `пр` (о ком?). Те же фильтры работают на `{{ Фамилия | … }}`,
  `{{ Имя | … }}`, `{{ Отчество | … }}`.
- **Инициалы.** `{{ ФИО.инициалы }}` → «Иванов П. С.»,
  `{{ ФИО.инициалы_впереди }}` → «П. С. Иванов»,
  `{{ ФИО.инициалы_слитно }}` → «Иванов П.С.» (удобно для имён файлов).
  В косвенных падежах склоняется фамилия: `{{ ФИО.инициалы | рд }}` →
  «Иванова П. С.».
- **Пол определяется сам** по отчеству, а если его нет — по имени.
  Неоднозначные имена (например «Саша» без отчества) можно задать вручную
  в `dyak.yaml`: `genders: { "Ким Саша": "ж" }`.
- **UPPERCASE сохраняется:** если в таблице фамилия записана капсом
  (`ИВАНОВ`), её склонения тоже капсом (`ИВАНОВА`).
- **Должности склоняются** пословно: `{{ Должность | рд }}` → «директора»,
  «главного бухгалтера», «инженера-программиста». Сложную фразу, которую
  движок берёт неидеально (например «заместитель генерального директора»
  в косвенных падежах), можно задать вручную через `overrides.position`
  в `dyak.yaml` (см. закомментированный пример) — ручные формы
  приоритетнее движка.
- В шаблоне теги пишутся **по заголовкам колонок**. Пробел в заголовке
  заменяется на подчёркивание: «Дата начала» → `{{ Дата_начала }}`.
  Если забыть и написать `{{ Дата начала }}` — dyak починит сам и
  напишет warning в лог.
- Добавить строки в `employees.xlsx` — на каждую получится отдельный
  документ.
- Поменять `--filename` — поменяется схема имён выходных файлов.
- Нестандартный заголовок (например «Сотрудник» вместо «Фамилия») —
  теперь распознаётся **по содержимому** без `aliases` (T016, см. раздел
  «Распознавание колонок по содержимому»). `aliases` в `dyak.yaml`
  остаётся как ручной override, если автоматика ошиблась.
- Заменить три колонки Фамилия/Имя/Отчество одной колонкой «ФИО» (как
  часто бывает в реальных таблицах): dyak сам разберёт её на части —
  причём колонку можно назвать и «ФИО», и «Сотрудник», и как угодно
  (распознавание по содержимому). В шаблоне будут доступны и целое
  `{{ ФИО }}`, и `{{ Фамилия }}` / `{{ Имя }}` / `{{ Отчество }}`. Если в
  ячейке два слова — отчество будет пустым.

## Важно (текущий этап)

Реализовано склонение **ФИО** (T002) и **должностей** (T003): фамилия/
имя/отчество, целое «ФИО» и должность склоняются по 6 падежам; есть
инициалы в трёх формах; пол определяется автоматически (колонка «Пол» не
нужна); сложные должности закрываются `overrides.position`.

Реализован и **старт с нуля** (T005): команда `dyak init` выкладывает
